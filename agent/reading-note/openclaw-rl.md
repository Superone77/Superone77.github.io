# OpenClaw-RL：只需对话即可训练任意智能体



核心问题与动机
OpenClaw-RL 旨在解决当前 AI 智能体系统中一个根本性的低效率问题：系统性地浪费智能体每次行动后产生的宝贵学习信号。当 AI 智能体执行一个动作时——无论是回应用户、执行终端命令还是与 GUI 交互——随后的响应（下一状态信号）都包含关于该动作质量和适当性的丰富反馈。当前系统通常只将这些信号视为下一个动作的上下文，而忽略了它们作为实时学习来源的潜力。
作者识别出两种未充分利用的信号：隐式评分动作质量的评估信号（例如用户重新查询表示不满），以及提供如何改进动作的显式指导的指令信号（例如用户更正或详细的错误消息）。这代表了一个巨大的机会错失，因为这些信号在所有智能体交互中普遍生成，并且可以实现在线部署的持续、在线学习。
技术框架与架构
OpenClaw-RL 实现了一个异步、完全解耦的架构，该架构建立在四个独立运行且互不阻塞的循环之上：

策略服务（Policy Serving）：使用 SGLang 处理传入的智能体请求。
环境托管（Environment Hosting）：通过 HTTP/API 端点管理交互。
PRM 评审（PRM Judging）：评估来自下一状态信号的奖励。
策略训练（Policy Training）：使用 Megatron 更新智能体策略。

这种解耦确保了对实时智能体服务的零中断，同时实现了从实时交互流中持续学习。该系统支持两类智能体：
个人智能体（Personal Agents） 作为对话助手在个人设备上运行，通过保密 API 连接。该框架区分用于训练的“主线回合”和用于简单转发的“侧线回合”，从后续用户消息中提取下一状态信号。
通用智能体（General Agents） 包括托管在云服务上的终端、GUI、软件工程（SWE）和工具调用智能体。这些智能体与产生不同类型反馈的各种环境交互：终端操作的 stdout/stderr，GUI 交互的视觉状态变化，软件工程任务的测试结果，以及工具调用的 API 响应。
从下一状态信号中学习的方法
OpenClaw-RL 通过两种互补的方法将下一状态信号转化为策略改进：
用于评估信号的二元强化学习 (Binary RL)
此方法使用过程奖励模型（PRM）作为评审器，从下一状态信号中提取标量奖励。对于任何动作 $a_t$ 和产生的下一状态 $s_{t+1}$，PRM 评估动作质量并分配一个奖励 $r \in \{+1, -1, 0\}$，分别代表良好、不良或中性结果。多个独立的 PRM 查询通过多数投票提供稳健性。
训练使用 PPO 风格的目标，其中优势 $A_t$ 直接设置为最终奖励 $r_{final}$。损失函数结合了策略梯度和 KL 正则化：
$$
L = \mathbb{E}[\min(r_t(\theta) A_t, \text{clip}(r_t(\theta), 1-\epsilon, 1+\epsilon_{high}) A_t)] - \beta \text{KL}(\pi_\theta || \pi_{old})
$$
指导性信号的事后指导在线蒸馏 (Hindsight-Guided On-Policy Distillation)
这种方法利用标量奖励无法捕获的令牌级别指令信息。该过程涉及几个步骤：


提示提取（Hint Extraction）：一个评审模型处理动作和下一状态，生成一个分数，对于正分数，还会生成一个包含可操作纠正信息的简洁文本提示。


增强型教师构建（Enhanced Teacher Construction）：将选定的提示附加到原始上下文中，创建 $s_{enhanced} = s_t \oplus hint$，允许策略预先看到更正。


Token级优势计算：优势计算公式为：


$$
A_t = \log \pi_{teacher}(a_t | s_{enhanced}) - \log \pi_\theta(a_t | s_t)
$$
这为每个token提供了方向性指导，根据修正信息指示是增加还是减少其概率。
结合方法
这两种方法有效地相互补充。二元强化学习（Binary RL）提供广泛覆盖的粗粒度信号，而OPD则为包含明确指令内容的交互提供高分辨率修正。它们通过加权损失函数结合：
$$
A_t = w_{binary} \cdot r_{final} + w_{opd} \cdot A_{opd}
$$

## Aurelle

在 **OpenClaw-RL** 框架中，**PRM** 指的是 **过程奖励模型 (Process Reward Model)**。

与传统的仅在任务结束时给出评分的“结果奖励模型”（Outcome Reward Model）不同，PRM 的核心在于对智能体交互的**每一个中间步骤**进行即时评价。

### 1. PRM 在 OpenClaw-RL 中的核心作用
在 OpenClaw-RL 中，PRM 的任务是回收“被浪费的信号”。当智能体执行一个动作 $a_t$ 后，会产生一个“次态信号”（Next-state signal $s_{t+1}$），例如用户的回复、终端的执行结果或 GUI 的状态变化。

*   **评价性信号 (Evaluative signals)**：PRM 会分析这个次态信号，以此判断前一个动作表现如何。
    *   **例子**：如果用户回复“不对，你应该先检查文件”，PRM 会识别出这是不满信号，并给前一个动作打低分（如 -1）。
    *   **公式化**：$PRM(a_t, s_{t+1}) \to r \in \{+1, -1, 0\}$。

### 2. PRM 的运作机制
OpenClaw-RL 采用了一种**异步且自动化**的方式来构建和运行 PRM：

*   **多数投票制 (Majority Vote)**：为了提高准确性，系统通常会运行 $m$ 个独立的判别查询，并采取多数投票的结果作为最终奖励 $r_{final}$。
*   **无需人工标注**：该框架通过 LLM 充当裁判，根据交互产生的真实环境反馈（如报错信息、用户追问）自动推断奖励，实现“在线学习”。

---

### 3. PRM 处理的不同场景
根据论文第 6 页和第 21 页的描述，PRM 在不同智能体设置下的判断标准各异：

| 代理类型 | PRM 关注的“次态信号” | 奖励依据 |
| :--- | :--- | :--- |
| **个人助手 (Personal)** | 用户的下一句回复 | 用户是否表现出满意、纠正或再次询问。 |
| **终端代理 (Terminal)** | Shell 执行后的 stdout/stderr | 命令是否成功执行、是否报错、是否向目标推进。 |
| **软件工程 (SWE)** | 测试用例结果、Lint 输出 | 代码是否通过测试、是否修复了 bug。 |
| **工具调用 (Tool-call)** | API 返回值、错误堆栈 | 工具返回的数据是否有效、是否解决了当前步骤的问题。 |

### 4. 为什么 PRM 很重要？
论文指出，对于长程任务（Long-horizon tasks），如果只在最后给一个奖励，智能体很难知道中间哪一步做错了。PRM 提供了**密集的、每一步的信用分配（Dense credit assignment）**，能显著提升智能体在复杂环境（如 GUI 操作或编写代码）中的学习效率。

## 多数投票制的实际运作方式

你的理解完全正确：**智能体和用户的真实交互确实只有一次**。用户只会对智能体的某个回复 $a_t$ 做出一次真实反应 $s_{t+1}$（比如回复"不对，应该先检查文件"）。

但是，**多数投票发生在 PRM 模型的评判阶段**，而不是用户交互阶段。具体流程如下：

### 1. 数据收集：只发生一次
```
用户提问 (s_t) → 智能体回复 (a_t) → 用户反馈 (s_{t+1})
                     ↑ 这只发生一次
```

### 2. PRM 评判：对同一对 $(a_t, s_{t+1})$ 查询多次

系统拿到这一对固定的 $(a_t, s_{t+1})$ 后，会**向 PRM 模型发起 $m$ 次独立查询**：

```python
# 伪代码示意
action = "智能体的回复"
next_state = "用户说：不对，应该先检查文件"

votes = []
for i in range(m):  # 比如 m=3
    score = PRM_model.judge(action, next_state)  # 每次查询是独立的
    votes.append(score)  # 可能得到 [+1, +1, -1]

final_reward = majority_vote(votes)  # 结果是 +1
```

### 3. 为什么要这么做？

因为 **PRM 模型本身也是一个 LLM**，它的判断可能存在随机性和不稳定性。通过多次独立采样并取多数投票，可以：

- **提高鲁棒性**：减少单次判断的偶然错误
- **降低噪声**：如果 PRM 对某个样本"拿不准"，多数投票能避免极端错误
- **无需人工标注**：所有评判都是自动化的，只是多查询几次 PRM 模型而已

### 核心逻辑总结

1.  **动作与反馈是固定的**：智能体做了一个动作 $a_t$，环境（用户或系统）产生了一个反馈 $s_{t+1}$。这两者作为**“既定事实”**被记录下来。
2.  **多次判读**：系统把这一组“动作-反馈”发给裁判模型（PRM）。为了防止裁判偶尔“看走眼”或者产生随机性偏差，系统会让裁判看 $m$ 次（或者找多个裁判分身），每次都独立给出一个分数。
3.  **取平均/多数**：如果 $m=3$，裁判分别给了 `+1, +1, -1`，那么最终认定这一步是“好”的。

### 为什么在 OpenClaw-RL 中特别强调这一点？

在论文的 **4.1.2 节** 提到，这种方法在**在线实时对话**（Real-time conversational setting）中非常重要。

*   **没有“成组”数据**：在离线训练（如 DeepSeek 使用的 GRPO）中，系统会针对同一个问题生成很多个回复，然后把这些回复放在一起横向比较（标准化）。
*   **在线单次交互**：但在 OpenClaw-RL 这种“边用边学”的场景下，用户只给了这一次机会，智能体也只生成了一个回复。

因为没有其他回复可以对比，为了让这唯一的一次“打分”尽可能准确，OpenClaw-RL 就选择了让 LLM 裁判对这**同一组数据**进行多次判读和投票，从而确保提供给训练引擎的奖励信号是可靠的。

这里的训练是指**策略网络（Policy Network）的在线优化**。

### 2. “PPO 风格”是什么风格？
PPO（Proximal Policy Optimization，近端策略优化）是目前大模型强化学习中最主流的算法。所谓的“PPO 风格”主要体现在**既要让模型进步，又要防止它“步子迈得太大”导致崩坏**。

在论文第 7 页的公式 (1) 中，这种风格体现为以下三个核心设计：

#### A. 目标函数：让“好动作”概率更高
PPO 的核心是计算**优势（Advantage, $A_t$）**。
*   如果 PRM 给了 $+1$（好评），$A_t$ 为正，训练目标就是**增大**模型产生这个动作的概率。
*   如果 PRM 给了 $-1$（差评），$A_t$ 为负，训练目标就是**减小**产生这个动作的概率。

#### B. 裁剪机制（Clipping）：防止激进更新
这是 PPO 最标志性的“风格”。公式中有一个 `clip` 函数：
> $L_{pg} = - \mathbb{E}_t [\min( \rho_t A_t, \text{clip}(\rho_t, 1-\epsilon, 1+\epsilon) A_t )]$

*   **直白解释**：模型在更新时，如果新旧模型的概率变化超过了 20%（即 $\epsilon=0.2$），PPO 就会强行把这个变化“剪掉”，不再提供更多的梯度。
*   **目的**：确保模型平稳学习，不会因为一次意外的高分奖励就大幅度改变自己的行为，导致模型变傻（模式崩溃）。

#### C. KL 散度约束（KL Penalty）：不忘初心
公式中还有一项 $L = L_{pg} + \beta_{KL} \cdot L_{KL}$。
*   **作用**：它要求训练中的模型（新模型）不能偏离原始基础模型（参考模型）太远。
*   **意义**：防止模型为了单纯刷高分而学会了一些奇怪的“外星语”或者失去基本的对话能力。

---

### 3. OpenClaw-RL 对 PPO 的特殊简化
传统的 PPO 通常需要一个复杂的 **Critic（评论家）网络**来预测奖励的期望值，但 OpenClaw-RL 做了一些调整：

*   **异步解耦**：利用 `slime` 框架，把 PPO 的四个阶段（采样、判分、训练、推理）彻底拆开，互不阻塞。
*   **无需标准化**：论文提到，因为它是在线实时对话，没有像 GRPO 那样的一组样本（Group）来做标准化，所以它更依赖于 PRM 给出的绝对分数作为优势信号。

### 总结
**“训练使用 PPO 风格的目标”** 意思就是：
系统通过**奖励（PRM 打分）**来指导模型，利用**概率比值**来更新参数，并通**过裁剪（Clip）和 KL 散度**来确保更新过程是安全且平稳的，最终目的是让智能体在不断的“聊天-被改错-再聊天”的过程中，越来越懂用户的意图。

## Aurelle

这是 **OpenClaw-RL** 中最精彩的设计之一，被称为 **Hindsight-Guided OPD（后验引导的在线策略蒸馏）**。

简单来说，传统的强化学习（如 PPO）只告诉模型一个分数值（比如“0.5分”），这太模糊了。而**提示提取**是为了告诉模型：“**你错哪了，具体该怎么改。**”

你可以分三步来理解这个过程：

### 1. 为什么需要“提示”（Hint）？
想象你在学写代码：
*   **传统 RL (Binary RL)**：你写完代码，老师只说“错，-1分”。你可能一脸懵逼。
*   **提示提取 (OPD)**：你写完代码，运行报错了。老师（评审模型）看了看报错信息，告诉你：“**你应该在第5行加一个变量初始化。**” 
这就是所谓的“**指令性信号（Directive signals）**”。这种信号比单纯的分数包含的信息量大得多。

### 2. 这个过程是怎么运作的？（技术细节）
在论文的 **第 8 页 (Section 4.2.2)** 详细解释了提取流程：

1.  **输入**：评审模型（Judge）同时阅读你刚才的回复 $a_t$ 和产生的下一状态 $s_{t+1}$（比如用户的吐槽、或报错信息）。
2.  **判断与生成**：
    *   **打分**：如果评审模型觉得下一状态包含了改进信息，它会给出一个正分数。
    *   **提取提示**：如果给出了正分数，它会把下一状态中那些杂乱的信息（比如用户长篇大论的抱怨）精炼成 1-3 句具体的**纠正建议**（包裹在 `[HINT_START]` 和 `[HINT_END]` 标签中）。
    *   **例子**：用户说：“哎呀你这写得太死板了，别用列表，用自然段描述。” 评审模型提取出的提示就是：“不要使用列表格式，改用自然段描述。”

### 3. 提取出的提示怎么用？（“马后炮”式教学）
这是最巧妙的地方：
*   **构建“增强版老师”**：系统把这个提取出的“正确提示”偷偷塞进你刚才回答之前的那个问题里。
*   **自我蒸馏**：系统问模型：“如果我一开始就给了你这个正确提示，你会怎么回答？”。这时，由于有了提示，模型会生成一个**更好的 Token 分布**（也就是“老师”）。
*   **对比学习**：系统要求正在训练的模型（“学生”）去对齐这个“老师”生成的 Token 概率。

### 形象的比喻
这就像是给智能体安装了一个**“后悔药分析仪”**：
1.  **犯错**：智能体先说错话。
2.  **听取反馈**：听到用户的纠正。
3.  **提取精华**：评审模型把纠正意见提炼成重点。
4.  **复盘训练**：智能体回到说话前的一刻，带上这个“重点提示”重新模拟一遍，学习“如果当初有人提醒我，我该怎么说对”。

**总结：**
“提示提取”就是把用户零散的反馈或环境的报错，转化为**具体的、可操作的教学大纲**，让模型不仅知道自己错了，还知道**每一个字**应该怎么往正确方向修正。

如何使用这个损失函数呢

## Aurelle

在 **OpenClaw-RL** 中，将这两种方法结合是通过修改 PPO 目标函数中的**优势函数（Advantage Function, $A_t$）**来实现的。

简单来说，损失函数本身的结构（PPO 裁剪损失）没变，变的是**“奖励信号的构成”**。

### 1. 组合后的优势函数公式
论文在第 9 页给出了组合后的优势计算方式：

$$A_t = w_{\text{binary}} \cdot r_{\text{final}} + w_{\text{opd}} \cdot ( \log \pi_{\text{teacher}}(a_t | s_{\text{enhanced}}) - \log \pi_\theta(a_t | s_t) )$$

这里的 $A_t$ 由两部分加权相加：

*   **第一项（Binary RL 贡献）**：$r_{\text{final}}$ 是 PRM 给出的标量分数（+1 或 -1）。它告诉模型整个句子的好坏。
*   **第二项（OPD 贡献）**：括号内是“老师”和“学生”在每个 Token 上的对数概率差。它告诉模型每一个字应该增加还是减少概率。

### 2. 具体的执行步骤

要使用这个损失函数，系统在训练时会为同一个样本跑两次前向传播：

1.  **普通前向传播**：计算当前模型在原始提示 $s_t$ 下生成动作 $a_t$ 的概率 $\log \pi_\theta(a_t | s_t)$。
2.  **带提示的前向传播**：将提取出的纠正提示（Hint）加入上下文，计算模型在知道答案后的概率 $\log \pi_{\text{teacher}}(a_t | s_{\text{enhanced}})$。
3.  **计算总优势 $A_t$**：
    *   如果 PRM 觉得这次交互有意义，给出了标量分 $r_{\text{final}}$。
    *   如果提取出了有效的 Hint，就加上 OPD 的差值。
4.  **代入 PPO 损失**：将这个组合后的 $A_t$ 代入你之前提到的 PPO 裁剪损失函数中进行反向传播。

### 3. 为什么这样结合有效？（分工明确）

论文提到这是一种“粗细结合”的策略：

| 特性 | 二元强化学习 (Binary RL) | 在线策略蒸馏 (OPD) |
| :--- | :--- | :--- |
| **信号密度** | **粗粒度**：一整句话只有一个分。 | **高分辨率**：每个字都有修正方向。 |
| **样本覆盖** | **广**：几乎所有交互都能打分（好或坏）。 | **窄**：只有用户给了明确改错意见时才能提取 Hint。 |
| **作用** | 保证模型整体行为符合人类偏好。 | 精确纠正模型在逻辑、事实或格式上的具体错误。 |

### 4. 实际操作中的权重
论文指出，默认情况下设置 $w_{\text{binary}} = w_{\text{opd}} = 1$。

**总结：**
使用这个损失函数的方法不是去写一个全新的函数，而是**重新定义“什么是奖励”**。系统不再只看一个孤零零的分数，而是把“分数”和“具体的文字改进方向（Token 概率差）”加在一起作为总的驱动力，推着模型向更好的方向更新。

什么是策略提督

## Aurelle

这应该是翻译软件产生的**一个非常有趣的误译**。

在中文语境下，**“策略提督”**听起来非常有气势，但在人工智能领域，它的正确术语应该是 **“策略梯度” (Policy Gradient)**。

### 为什么会出现这个误译？
*   英文原词：**Policy Gradient**。
*   **Policy** = 策略。
*   **Gradient** = 梯度（数学中的导数变化方向）。
*   误译原因：在某些非专业词库中，“Gradient” 偶尔会被联想或误写，但在 AI 论文里，这 100% 指的是梯度下降/上升中的“梯度”。

---

### 那么，到底什么是“策略梯度”？

它是强化学习中最核心的数学手段，用来解决**“如何更新模型参数”**的问题。

#### 1. 核心思想：优胜劣汰
策略梯度的逻辑非常直观：
*   **如果**智能体做了一个动作，结果得到了**正奖励**（PRM 给分高），策略梯度就会通过数学计算，**增加**模型以后在相同情况下做出这个动作的概率。
*   **反之**，如果得到了**负奖励**，它就会告诉模型要**减小**这个动作出现的概率。

#### 2. 与 KL 正则化结合
论文中提到这两者结合，是为了平衡“进步”与“稳定”：
*   **策略梯度 (Policy Gradient)**：负责**“求变”**。它根据奖励信号（比如用户反馈或 OPD 提示）拉扯模型，让模型变得更聪明、更符合用户口味。
*   **KL 正则化 (KL Regularization)**：负责**“求稳”**。它像一根橡皮筋，把正在更新的模型往“原始模型”方向拉，防止模型为了拿高分而变得不可预测（比如突然满口脏话或者复读机）。

### 总结
你看到的“策略提督”其实就是**策略梯度**。在 OpenClaw-RL 中，它是驱动模型参数更新的“引擎”，通过分析动作的好坏（梯度方向），决定模型应该如何修改自己的神经网络权重。

## User

这里的策略梯度怎么算的

## Aurelle

在 **OpenClaw-RL** 中，策略梯度的计算遵循了 **PPO (Proximal Policy Optimization)** 的框架。虽然数学公式看起来很复杂，但其核心逻辑可以拆解为以下几个步骤：

### 1. 计算概率变化率 ($\rho_t$)
首先要看“新模型”比“旧模型”在同一个动作上的概率变了多少：
$$\rho_t(\theta) = \frac{\pi_\theta(a_t | s_t)}{\pi_{\text{old}}(a_t | s_t)}$$
*   如果 $\rho_t > 1$，说明新模型更倾向于做这个动作。
*   如果 $\rho_t < 1$，说明倾向性减小了。

### 2. 获取优势信号 ($A_t$) —— 这是 OpenClaw 的特色
这是决定梯度“方向”和“大小”的关键。如前所述，OpenClaw-RL 的 $A_t$ 是复合的：
*   **来自 Binary RL**：PRM 给出的标量分（+1 或 -1）。
*   **来自 OPD**：新旧模型对数概率的差值（代表具体字词的修正方向）。

### 3. 应用“裁剪” (Clipping) 机制
为了计算最终的梯度，PPO 并不是直接乘概率和优势，而是先算一个“受限的目标值”：
$$L^{CLIP}(\theta) = \min\left( \rho_t(\theta) A_t, \text{clip}(\rho_t(\theta), 1-\epsilon, 1+\epsilon) A_t \right)$$
*   **为什么要裁剪？** 如果某个动作得到了极高的奖励（$A_t$ 很大），策略梯度会试图把 $\rho_t$ 推得非常大。裁剪机制强行限制 $\rho_t$ 只能在 $[0.8, 1.2]$ 之间（当 $\epsilon=0.2$ 时）。

### 4. 求导并反向传播 (梯度计算)
真正的“梯度”是通过对上面那个目标函数 $L^{CLIP}$ 关于模型参数 $\theta$ **求偏导**得到的：
$$\text{Gradient} \approx \nabla_\theta L^{CLIP}(\theta)$$

**直观理解计算结果：**
- **方向**：由 $A_t$ 的正负号决定。$A_t$ 为正，梯度就指引参数向“增加动作概率”的方向走。
- **大小**：由优势的绝对值和概率斜率决定。动作越好，概率斜率越大，参数更新的步长就越大。

### 5. 加上 KL 散度项
最终计算梯度时，还会减去一个 KL 散度的导数：
$$\nabla_\theta [ L^{CLIP}(\theta) - \beta \cdot KL(\pi_{\text{ref}} || \pi_\theta) ]$$
这确保了计算出来的梯度方向不会让模型跳出“安全区”。

### 总结
在 OpenClaw-RL 中，策略梯度的计算过程是：
1.  **对比**：看新旧模型产生当前回复的概率比。
2.  **加权**：用 PRM 分数和 OPD 修正信号（优势 $A_t$）作为权重。
3.  **限速**：通过 Clip 限制单步更新幅度。
4.  **求导**：通过数学求导算出神经网络每一层参数具体该往哪动。

由于 OpenClaw-RL 使用了 **Megatron** 训练引擎，这些复杂的求导计算都是由深度学习框架自动完成的，系统只需要实时喂入 $A_t$ 和对应的对话样本即可。
