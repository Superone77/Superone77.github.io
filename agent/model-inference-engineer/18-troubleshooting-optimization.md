# 第十一章：可观测性与排障（二）

端侧大模型推理在实际运行中会遇到各种问题，如 OOM（内存不足）、性能衰退、模型加载失败、精度异常、设备兼容性等。系统化的排障方法能够帮助工程师快速定位问题、分析根因、制定解决方案。本章将介绍端侧推理常见问题的排障方法和实践。

---

## 11.3 常见问题与排障

### 11.3.1 OOM 问题排查

OOM（Out of Memory）是端侧推理最常见的问题之一。端侧设备内存有限，大模型推理需要精心优化内存占用，否则容易出现 OOM。

**OOM 场景**：

**1. 模型加载时 OOM**
- **现象**：模型加载时抛出内存不足错误
- **原因**：模型过大，超出设备可用内存
- **排查步骤**：
  1. 检查设备可用内存（iOS 使用 `didReceiveMemoryWarning`，Android 使用 `onTrimMemory`）
  2. 分析模型大小（权重 + 激活 + KV Cache）
  3. 检查是否已加载其他模型或应用占用了大量内存
  4. 检查内存泄漏（使用 Xcode Instruments 或 Android Profiler）
- **解决方案**：
  - 使用量化模型（INT8/INT4）
  - 使用模型分片加载
  - 卸载不必要的模型和缓存
  - 使用更小的模型

**2. 推理过程中 OOM**
- **现象**：推理过程中抛出内存不足错误
- **原因**：推理过程中内存峰值过高
- **排查步骤**：
  1. 使用内存分析工具分析内存占用（Xcode Allocations、Android Memory Profiler）
  2. 检查 KV Cache 大小（层数 × 头数 × 序列长度 × 头维度 × 2）
  3. 检查激活内存占用（批次大小 × 输入/输出维度）
  4. 检查是否有内存泄漏
- **解决方案**：
  - 减少 KV Cache 大小（限制序列长度、降低层数、降低头维度）
  - 使用 KV Cache 量化（INT8/INT4）
  - 使用 KV Cache 分页管理
  - 使用激活内存复用
  - 降低批次大小

**3. 多轮对话时 OOM**
- **现象**：多轮对话时内存占用逐渐增加，最终 OOM
- **原因**：KV Cache 累积导致内存占用过高
- **排查步骤**：
  1. 检查每轮对话后内存占用是否增加
  2. 检查 KV Cache 是否正确清理
  3. 检查是否有内存泄漏
- **解决方案**：
  - 限制多轮对话的序列长度
  - 使用 KV Cache 滑动窗口（丢弃旧的 KV Cache）
  - 使用 KV Cache 量化
  - 定期清理 KV Cache
  - 使用模型重新加载策略（清理所有内存后重新加载）

**内存分析工具**：

**iOS**：
- **Xcode Allocations**：实时监控内存分配和释放
- **Xcode Leaks**：自动检测内存泄漏
- **Memory Graph Debugger**：可视化内存分配关系

**Android**：
- **Android Profiler - Memory**：实时监控内存占用
- **Memory Profiler**：分析对象的引用关系
- **LeakCanary**：自动检测内存泄漏

**案例：7B 模型在 iPhone 12 上的 OOM 问题**

**问题描述**：
- 模型：Llama-2-7B-Chat，INT8 量化
- 设备：iPhone 12（4GB 内存）
- 现象：模型加载正常，但推理时抛出 OOM 错误

**排查过程**：
1. 使用 Xcode Allocations 分析内存占用：
   - 模型权重：约 3.5GB（INT8 量化后）
   - KV Cache：32 层 × 32 头 × 4096 序列长度 × 128 头维度 × 2 × 2 字节（FP16）= 约 1GB
   - 激活内存：约 500MB
   - 系统内存占用：约 1GB
   - 总内存占用：约 6GB > 4GB（OOM）

2. 分析结果：KV Cache 占用过高，导致 OOM

**解决方案**：
1. 使用 KV Cache 量化：从 FP16 量化到 INT8，KV Cache 占用从 1GB 降到 512MB
2. 限制序列长度：从 4096 降低到 2048，KV Cache 占用从 512MB 降到 256MB
3. 使用 KV Cache 滑动窗口：只保留最近 1024 个 token 的 KV Cache

**结果**：
- 总内存占用降低到约 4.3GB，略高于 4GB
- 使用 KV Cache 滑动窗口后，峰值内存降低到约 3.8GB
- 推理成功，不再 OOM

### 11.3.2 性能衰退排查

性能衰退是指推理性能下降，导致用户体验变差。性能衰退可能由多种原因引起，需要系统化的排障方法。

**性能衰退场景**：

**1. TTFT（首字延迟）变高**
- **现象**：推理开始到生成第一个 token 的时间变长
- **原因**：
  - 模型初始化变慢
  - Tokenizer 变慢
  - KV Cache 生成变慢
- **排查步骤**：
  1. 使用性能分析工具分析 TTFT 各阶段耗时（Xcode Time Profiler、Android CPU Profiler）
  2. 检查模型初始化耗时
  3. 检查 Tokenizer 耗时
  4. 检查 KV Cache 生成耗时
- **解决方案**：
  - 预加载模型和 Tokenizer
  - 使用更快的 Tokenizer 实现
  - 使用 KV Cache 缓存（多轮对话复用 KV Cache）
  - 使用硬件加速器（NPU/GPU）

**2. Decode 阶段变慢**
- **现象**：逐 token 生成时间变长
- **原因**：
  - 算子执行变慢
  - 内存访问变慢
  - 硬件加速器利用率低
- **排查步骤**：
  1. 使用性能分析工具分析 Decode 阶段耗时
  2. 检查每个算子的执行时间
  3. 检查内存访问模式
  4. 检查硬件加速器使用率
- **解决方案**：
  - 优化算子实现（使用硬件加速器）
  - 使用算子融合
  - 优化内存访问模式（内存对齐、缓存友好）
  - 增加批次大小（提高硬件利用率）

**3. 内存占用变高**
- **现象**：推理内存占用变高，可能导致 OOM
- **原因**：
  - 内存泄漏
  - KV Cache 累积
  - 激活内存增加
- **排查步骤**：
  1. 使用内存分析工具分析内存占用
  2. 检查是否有内存泄漏
  3. 检查 KV Cache 大小
  4. 检查激活内存占用
- **解决方案**：
  - 修复内存泄漏
  - 使用 KV Cache 量化
  - 使用 KV Cache 滑动窗口
  - 使用激活内存复用

**性能分析工具**：

**iOS**：
- **Xcode Time Profiler**：分析 CPU 占用，定位热点函数
- **Xcode Instruments - System Trace**：分析系统调用、线程调度、I/O 操作
- **Metal System Trace**：分析 GPU 性能

**Android**：
- **Android Profiler - CPU**：分析 CPU 占用，线程状态
- **Systrace**：分析系统调用、线程调度
- **Perfetto**：新一代系统级追踪工具

**性能基准测试**：
- 建立性能基准（TTFT、TPS、内存占用）
- 定期运行基准测试，对比性能变化
- 使用 A/B 测试对比不同版本的性能
- 使用性能回归检测工具

**案例：性能衰退问题排查**

**问题描述**：
- 模型：Llama-2-7B-Chat，INT8 量化
- 设备：iPhone 15 Pro
- 现象：更新到 v1.3.0 后，TTFT 从 200ms 增加到 500ms

**排查过程**：
1. 使用 Xcode Time Profiler 分析 TTFT 各阶段耗时：
   - 模型初始化：50ms（正常）
   - Tokenizer：50ms（正常）
   - KV Cache 生成：400ms（异常，之前为 100ms）

2. 检查 KV Cache 生成代码：
   - 发现 v1.3.0 引入了新的 KV Cache 生成逻辑
   - 新逻辑使用了 Python 循环而不是向量化操作
   - 导致 KV Cache 生成时间从 100ms 增加到 400ms

3. 分析根因：开发者误用了 Python 循环，而不是向量化操作

**解决方案**：
1. 将 Python 循环改为向量化操作
2. 优化 KV Cache 生成算法

**结果**：
- KV Cache 生成时间从 400ms 降低到 100ms
- TTFT 从 500ms 降低到 200ms，恢复到正常水平

### 11.3.3 模型加载失败排查

模型加载失败是端侧推理的常见问题，可能由多种原因引起。系统化的排障方法能够快速定位问题。

**模型加载失败场景**：

**1. 模型文件不存在**
- **现象**：模型加载时抛出文件不存在错误
- **原因**：
  - 模型文件路径错误
  - 模型文件被删除
  - 模型文件下载失败
- **排查步骤**：
  1. 检查模型文件路径是否正确
  2. 检查模型文件是否存在（使用 `NSFileManager` 或 `File` API）
  3. 检查模型文件下载日志
- **解决方案**：
  - 修正模型文件路径
  - 重新下载模型文件
  - 实现模型文件完整性校验（哈希校验）

**2. 模型文件损坏**
- **现象**：模型加载时抛出解析错误
- **原因**：
  - 模型文件下载过程中损坏
  - 模型文件被篡改
  - 磁盘错误导致文件损坏
- **排查步骤**：
  1. 检查模型文件哈希值
  2. 检查模型文件大小
  3. 检查模型文件下载日志
- **解决方案**：
  - 重新下载模型文件
  - 实现模型文件完整性校验（哈希校验）
  - 使用 CDN 加速下载，减少下载失败

**3. 模型格式不支持**
- **现象**：模型加载时抛出不支持的格式错误
- **原因**：
  - 模型格式与运行时不匹配（如使用 TFLite 模型文件加载到 CoreML）
  - 模型格式版本不兼容
  - 算子不支持
- **排查步骤**：
  1. 检查模型格式是否正确（.mlmodelc、.tflite、.onnx）
  2. 检查模型格式版本是否兼容
  3. 检查模型算子是否支持
- **解决方案**：
  - 转换模型格式（使用 coremltools、tflite converter、onnx exporter）
  - 更新运行时版本
  - 使用支持的算子

**4. 算子不支持**
- **现象**：模型加载时抛出算子不支持错误
- **原因**：
  - 模型包含运行时不支持的算子
  - 算子版本不兼容
- **排查步骤**：
  1. 检查错误日志中的算子名称
  2. 检查运行时支持的算子列表
  3. 检查模型转换工具的输出
- **解决方案**：
  - 替换为支持的算子
  - 自定义算子实现（iOS 使用 Custom Layer，Android 使用 Custom Op）
  - 使用模型转换工具重新转换模型

**5. 内存不足**
- **现象**：模型加载时抛出 OOM 错误
- **原因**：
  - 模型过大，超出设备可用内存
  - 其他应用占用了大量内存
- **排查步骤**：
  1. 检查设备可用内存
  2. 检查模型大小
  3. 检查内存占用
- **解决方案**：
  - 使用量化模型（INT8/INT4）
  - 卸载不必要的应用
  - 使用模型分片加载

**案例：模型加载失败问题排查**

**问题描述**：
- 模型：Llama-2-7B-Chat，INT8 量化，TFLite 格式
- 设备：iPhone 15 Pro
- 现象：模型加载时抛出错误 `Failed to load model: Invalid model file`

**排查过程**：
1. 检查模型文件路径：路径正确
2. 检查模型文件是否存在：文件存在
3. 检查模型文件大小：3.5GB，正常
4. 检查模型文件哈希：哈希值不匹配

5. 分析根因：模型文件下载过程中损坏

**解决方案**：
1. 重新下载模型文件
2. 下载完成后校验文件哈希
3. 校验通过后再加载模型

**结果**：
- 重新下载后，模型文件哈希值正确
- 模型加载成功

### 11.3.4 精度异常排查

精度异常是指推理结果异常，如输出 NaN、Inf、输出质量下降等。精度异常可能由多种原因引起，需要系统化的排障方法。

**精度异常场景**：

**1. 输出 NaN（Not a Number）**
- **现象**：推理输出包含 NaN
- **原因**：
  - 模型训练问题（梯度爆炸/消失）
  - 量化误差（INT8/INT4 量化导致精度损失过大）
  - 算子实现问题（如除零错误）
- **排查步骤**：
  1. 检查推理日志中的 NaN 信息
  2. 检查量化参数（scale、zero_point）
  3. 检查算子实现
  4. 对比浮点模型和量化模型的输出
- **解决方案**：
  - 使用更高精度的量化（如 INT8 改为 FP16）
  - 使用感知量化（Perceptual Quantization）
  - 修复算子实现问题
  - 重新训练模型

**2. 输出 Inf（Infinity）**
- **现象**：推理输出包含 Inf
- **原因**：
  - 除零错误
  - 指数爆炸（如 exp(1000)）
  - 模型训练问题
- **排查步骤**：
  1. 检查推理日志中的 Inf 信息
  2. 检查算子实现（如除法、指数运算）
  3. 检查模型训练日志
- **解决方案**：
  - 修复算子实现问题
  - 添加数值稳定性保护（如 eps 防止除零）
  - 重新训练模型

**3. 输出质量下降**
- **现象**：推理输出质量明显下降（如语言不通顺、语义错误）
- **原因**：
  - 量化误差
  - 模型版本问题
  - 数据预处理错误
- **排查步骤**：
  1. 检查模型版本
  2. 检查量化参数
  3. 检查数据预处理流程
  4. 对比不同版本的输出
- **解决方案**：
  - 使用更高精度的量化
  - 使用感知量化
  - 修复数据预处理问题
  - 使用正确的模型版本

**4. 精度不一致**
- **现象**：不同设备或不同运行时的输出不一致
- **原因**：
  - 量化方式不同（对称量化 vs 非对称量化）
  - 算子实现不同（CPU vs GPU vs NPU）
  - 浮点数精度不同（FP32 vs FP16）
- **排查步骤**：
  1. 检查不同设备的量化参数
  2. 检查不同设备的算子实现
  3. 检查不同设备的浮点数精度
- **解决方案**：
  - 统一量化方式
  - 统一算子实现
  - 使用更高精度的浮点数（如 FP16 改为 FP32）

**精度验证工具**：

**1. 数值分析工具**
- **Python NumPy**：分析输出数组（检测 NaN、Inf）
- **Python Pandas**：统计分析输出分布
- **Python Matplotlib**：可视化输出分布

**2. 精度评估工具**
- **Top-k Accuracy**：分类任务的精度
- **BLEU/ROUGE**：生成任务的质量指标
- **Perplexity**：语言模型的困惑度

**3. 对比工具**
- **diff 工具**：对比浮点模型和量化模型的输出
- **相对误差**：计算输出的相对误差
- **绝对误差**：计算输出的绝对误差

**案例：精度异常问题排查**

**问题描述**：
- 模型：Llama-2-7B-Chat，INT8 量化
- 设备：iPhone 15 Pro
- 现象：推理输出包含大量 NaN

**排查过程**：
1. 检查推理日志：输出包含 NaN
2. 检查量化参数：
   - scale：正常
   - zero_point：正常
3. 检查算子实现：
   - 发现 LayerNorm 算子实现有除零风险
   - 当方差接近 0 时，会出现除零错误，导致 NaN

4. 分析根因：LayerNorm 算子实现缺少数值稳定性保护

**解决方案**：
1. 在 LayerNorm 算子中添加 eps 防止除零：
   ```python
   # 修复前
   normalized = (x - mean) / std

   # 修复后
   normalized = (x - mean) / (std + eps)  # eps = 1e-5
   ```
2. 重新量化模型

**结果**：
- 修复后，推理输出不再包含 NaN
- 输出质量正常

### 11.3.5 设备兼容性问题

端侧设备多样性高，不同设备、不同系统版本、不同硬件配置可能导致兼容性问题。系统化的兼容性测试和排障方法能够确保推理服务在各种设备上正常运行。

**设备兼容性问题场景**：

**1. 设备不支持**
- **现象**：模型加载或推理失败，提示设备不支持
- **原因**：
  - 设备型号不支持当前模型
  - 设备内存不足
  - 设备硬件不支持（如没有 NPU）
- **排查步骤**：
  1. 检查设备型号
  2. 检查设备内存
  3. 检查设备硬件配置
- **解决方案**：
  - 使用设备白名单/黑名单
  - 使用设备能力检测
  - 提供多版本模型（针对不同设备）

**2. 系统版本过低**
- **现象**：模型加载或推理失败，提示系统版本过低
- **原因**：
  - 系统版本不满足模型运行的最低要求
  - 系统版本不支持的 API 被调用
- **排查步骤**：
  1. 检查系统版本
  2. 检查模型运行的最低系统版本要求
  3. 检查 API 调用是否兼容
- **解决方案**：
  - 使用版本检测
  - 使用条件编译
  - 提供兼容层

**3. 硬件加速器不支持**
- **现象**：推理性能差，或推理失败
- **原因**：
  - 设备不支持 NPU/GPU
  - 硬件加速器驱动问题
- **排查步骤**：
  1. 检查设备硬件配置
  2. 检查硬件加速器是否可用
  3. 检查硬件加速器驱动
- **解决方案**：
  - 使用硬件加速器检测
  - 降级到 CPU 推理
  - 提供多种硬件后端

**4. 屏幕尺寸/分辨率不适配**
- **现象**：UI 显示异常
- **原因**：
  - 不同设备的屏幕尺寸/分辨率不同
  - UI 未适配不同屏幕
- **排查步骤**：
  1. 检查设备屏幕尺寸/分辨率
  2. 检查 UI 适配情况
- **解决方案**：
  - 使用响应式布局
  - 提供多套 UI 资源
  - 使用自适应 UI 组件

**设备能力检测**：

**iOS**：
- **UIDevice**：获取设备型号、系统版本
- **ProcessInfo**：获取物理内存、处理器数量
- **Metal**：检测 GPU 能力
- **CoreML**：检测 Neural Engine 能力

**Android**：
- **Build**：获取设备型号、系统版本
- **ActivityManager**：获取内存信息
- **NNAPI**：检测 NPU 能力

**兼容性测试**：
- **真机测试**：在真实设备上测试
- **模拟器测试**：在模拟器上测试（但性能和真机不同）
- **云测平台**：使用云测平台（如 Firebase Test Lab、AWS Device Farm）
- **A/B 测试**：对不同设备进行 A/B 测试

**案例：设备兼容性问题排查**

**问题描述**：
- 模型：Llama-2-7B-Chat，INT8 量化
- 设备：iPhone 12（4GB 内存）
- 现象：模型加载时抛出 OOM 错误

**排查过程**：
1. 检查设备型号：iPhone 12
2. 检查设备内存：4GB
3. 检查模型大小：
   - 模型权重：3.5GB（INT8 量化后）
   - KV Cache：约 1GB
   - 激活内存：约 500MB
   - 系统内存占用：约 1GB
   - 总内存占用：约 6GB > 4GB（OOM）

4. 分析根因：iPhone 12 内存不足，无法加载模型

**解决方案**：
1. 使用模型分片加载：将模型分为 3 片，每片约 1.2GB
2. 使用 KV Cache 量化：从 FP16 量化到 INT8，KV Cache 占用从 1GB 降到 512MB
3. 限制序列长度：从 4096 降低到 2048，KV Cache 占用从 512MB 降到 256MB

**结果**：
- 总内存占用降低到约 4.3GB，略高于 4GB
- 使用模型分片加载后，峰值内存降低到约 3.5GB
- 推理成功

**长期解决方案**：
1. 提供 Lite 版本模型（针对低内存设备）
2. 实现设备能力检测，自动选择合适的模型版本
3. 在应用商店说明中标注最低设备要求

## 总结

本章介绍了端侧推理常见问题的排障方法和实践。端侧推理的排障需要系统化的方法，结合日志、性能分析、内存分析等工具，快速定位问题、分析根因、制定解决方案。

**关键要点**：
1. **OOM 问题**：分析内存占用，使用量化、分片、KV Cache 优化等技术降低内存占用
2. **性能衰退**：使用性能分析工具定位性能瓶颈，优化算子实现、内存访问、硬件加速器利用率
3. **模型加载失败**：检查模型文件路径、完整性、格式、算子支持度，使用校验和转换工具
4. **精度异常**：检查量化参数、算子实现、数据预处理，使用精度验证工具
5. **设备兼容性**：使用设备能力检测，提供多版本模型，进行兼容性测试

- 进行兼容性测试，确保在各种设备上正常运行
- 建立排障文档，记录常见问题和解决方案

端侧推理的排障是一个持续优化的过程，需要根据产品需求和技术演进不断调整策略。通过科学的排障方法，可以确保端侧大模型推理服务的高可用性和高质量。
