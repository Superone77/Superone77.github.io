# 端侧模型格式与转换

端侧推理的核心挑战在于**模型格式碎片化**：不同平台、不同框架、不同硬件都有各自支持的模型格式。理解常见端侧模型格式、转换工具链和选型权衡，是端侧推理工程师的基础能力。

---

## 常见端侧模型格式

### ONNX：跨平台中间格式

ONNX（Open Neural Network Exchange）是微软和 Facebook（Meta）主导开发的跨平台神经网络模型交换格式。ONNX 的核心定位是**中间表示**，而非推理运行时，这使得它成为不同框架之间的桥梁。

ONNX 的核心特性：

- **跨框架支持**：支持 PyTorch、TensorFlow、Keras、MXNet 等主流框架的模型导出
- **跨平台推理**：ONNX Runtime、TensorFlow Lite、CoreML 等运行时都支持 ONNX 格式
- **版本控制（Opset）**：通过 Opset 版本控制算子集合，保证兼容性
- **生态成熟**：有丰富的转换工具（onnx-tensorflow、torch.onnx.export）和优化工具（ONNX-TensorRT）

ONNX 在端侧推理中的定位是**中间层**：通常不会直接在端侧使用 ONNX 格式推理，而是先转换为 TFLite/CoreML/NCNN 等目标格式，再部署到端侧。ONNX 的价值在于提供统一的中间表示，简化跨框架转换流程。

### TFLite / LiteRT：Android 官方格式

TFLite（TensorFlow Lite）是 Google 官方的端侧推理格式，专为 Android 和嵌入式设备设计。2025 年，Google 推出了 LiteRT 作为 TFLite 的下一代框架，增加了对大语言模型（LLM）的优化支持。

TFLite/LiteRT 的核心特性：

- **跨平台支持**：支持 Android、iOS、Linux、嵌入式设备
- **硬件加速**：通过 Delegate 机制支持 GPU、NPU、DSP 等硬件后端
- **多框架转换**：支持 PyTorch、JAX、TensorFlow 到 TFLite 的转换
- **模型优化**：内置量化、剪枝、算子融合等优化工具
- **生态成熟**：有丰富的预训练模型和社区支持

TFLite 在端侧推理中的定位是**Android 端首选格式**：对于 Android 平台，TFLite 是官方推荐格式，有成熟的工具链和社区支持。LLM 场景下，LiteRT-LM 专门针对生成式模型优化，支持 KV Cache 优化、投机采样等端侧 LLM 特性。

### CoreML：iOS 专属格式

CoreML 是 Apple 官方的端侧推理格式，专为 iOS、macOS、tvOS、watchOS 设计。CoreML 与 Apple 生态深度集成，提供统一的模型表示和推理 API。

CoreML 的核心特性：

- **平台专属**：只支持 Apple 平台（iOS、macOS、tvOS、watchOS、visionOS）
- **硬件优化**：自动选择 CPU、GPU、Neural Engine 最优计算设备
- **统一内存架构**：模型参数常驻内存，数据移动成本低
- **模型加密**：支持模型加密和签名，防止逆向工程
- **端侧个性化**：支持设备上微调（On-device Personalization）

CoreML 在端侧推理中的定位是**iOS 端首选格式**：对于 Apple 平台，CoreML 是官方推荐格式，有完善的工具链（coremltools）和 API（Swift/Objective-C）。

### MNN、NCNN、GGUF：国产与专用格式

除了主流格式外，端侧推理还有一些国产或专用格式：

**MNN（Mobile Neural Network）**
- 阿里巴巴开源，专为移动端优化
- 支持跨平台（Android、iOS、Linux）
- 算子覆盖度高，推理性能优于 TFLite 在某些场景
- 适合需要极致优化的国产手机端部署

**NCNN（Neural Computer Network）**
- 腾讯开源，专为移动端优化
- 支持跨平台（Android、iOS、Linux）
- 针对 ARM 架构深度优化，内存占用低
- 适合需要极致优化的移动端视觉应用

**GGUF（GPT-Generated Unified Format）**
- llama.cpp 社区推出的端侧 LLM 格式
- 支持多种量化格式（FP16、INT8、INT4、Q4_K_M 等）
- 内存映射（Memory-mapped）加载，支持超大模型
- 适合端侧 LLM 推理（如 llama.cpp、ollama）

### 模型格式对比表

下表总结了主要端侧模型格式在端侧推理中的定位、优势与限制：

| 维度 | ONNX | TFLite/LiteRT | CoreML | MNN/NCNN | GGUF |
|------|-------|----------------|---------|-----------|------|
| **核心定位** | 跨平台中间格式 | Android 官方格式 | iOS 官方格式 | 国产移动端优化 | 端侧 LLM 专用格式 |
| **平台支持** | 全平台（跨框架） | 全平台 | Apple 平台 | 跨平台 | 跨平台 |
| **硬件加速** | 取决于运行时 | GPU、NPU、DSP | CPU、GPU、Neural Engine | CPU、GPU、NEON | CPU、GPU |
| **模型优化** | 依赖外部工具 | 内置量化、剪枝 | 内置优化 | 内置优化 | 内置量化 |
| **转换成本** | 低（主流框架都支持） | 中（需转换到 TFLite） | 中（需转换到 CoreML） | 中（需转换到 MNN/NCNN） | 低（LLaMA 模型直接转换） |
| **维护成本** | 低（生态成熟） | 低（Google 官方支持） | 低（Apple 官方支持） | 中（社区支持） | 中（社区支持） |
| **推理性能** | 中（依赖运行时） | 高（针对 Android 优化） | 高（针对 Apple 优化） | 高（针对 ARM 优化） | 高（针对 LLM 优化） |
| **算子覆盖度** | 高（标准算子齐全） | 高（针对优化算子） | 中（Apple 生态） | 高（移动端算子） | 低（LLM 特定） |
| **内存占用** | 中（未优化） | 低（内置优化） | 低（内置优化） | 低（深度优化） | 低（量化+内存映射） |
| **LLM 支持** | 有（需额外优化） | 有（LiteRT-LM） | 有（Metal Shaders） | 有（但不如 GGUF） | 最佳（专用格式） |
| **典型场景** | 跨框架转换、中间层 | Android 推理 | iOS 推理 | 国产手机推理 | 端侧 LLM 推理 |

**工程启示**：

1. **跨平台项目选 ONNX 作为中间层**：如果需要支持多平台（iOS + Android），建议先用 ONNX 作为中间格式，再分别转换为 TFLite（Android）和 CoreML（iOS）。
2. **单一平台选官方格式**：如果只需要支持 Android 或 iOS，直接选 TFLite 或 CoreML，避免不必要的转换成本。
3. **LLM 场景考虑 GGUF**：如果重点是端侧 LLM 推理，GGUF + llama.cpp 生态是成熟方案，支持多种量化格式和超大模型。
4. **国产手机考虑 MNN/NCNN**：如果目标是国产手机（华为、小米、OPPO、vivo），MNN/NCNN 可能有更好的优化效果。

## 模型转换工具链

### 转换工具概览

模型转换是端侧推理的必经之路：从训练框架（PyTorch、TensorFlow）到端侧格式（TFLite、CoreML、MNN）。常见的转换工具链：

**PyTorch → ONNX**
- 工具：`torch.onnx.export`
- 优点：PyTorch 官方支持，转换质量高
- 缺点：部分自定义算子可能无法导出
- 注意事项：需要设置动态维度（如 batch_size = -1），确保模型可导出

**ONNX → TFLite**
- 工具：`onnx-tflite` 或 `tf-lite-converter`
- 优点：ONNX 转到 TFLite 的成熟方案
- 缺点：部分 ONNX 算子可能不被 TFLite 支持
- 注意事项：需要检查算子兼容性，必要时做算子替换

**PyTorch/TensorFlow → CoreML**
- 工具：`coremltools`
- 优点：Apple 官方工具，支持 PyTorch、TensorFlow
- 缺点：转换失败时调试困难
- 注意事项：需要设置正确的输入输出类型（如 `image_type`）

**PyTorch → GGUF**
- 工具：`llama.cpp` 的 `convert.py`
- 优点：针对 LLM 优化，支持多种量化格式
- 缺点：只支持 LLaMA 系列模型
- 注意事项：需要将模型转换为特定格式（如 GGML → GGUF）

### 转换流程与注意事项

典型的端侧模型转换流程：

1. **导出到 ONNX**：从 PyTorch/TensorFlow 导出到 ONNX 中间格式
2. **验证 ONNX 模型**：使用 `onnx.checker` 验证模型正确性
3. **转换到目标格式**：从 ONNX 转到 TFLite/CoreML/MNN
4. **量化与优化**：对模型进行量化（INT8、INT4）和算子融合
5. **部署测试**：在目标设备上测试模型性能和精度

**注意事项**：

- **动态维度 vs 静态维度**：端侧推理通常需要静态维度（如固定 batch_size），需要在转换时设置
- **算子兼容性**：部分自定义算子可能不被目标格式支持，需要算子替换或自定义实现
- **版本控制**：ONNX Opset 版本、TFLite 版本、CoreML 版本都会影响转换结果
- **精度损失**：量化（INT8、INT4）会带来精度损失，需要校准（Calibration）和精度验证

### 转换过程中的精度损失

模型转换可能带来精度损失，主要来源：

1. **算子实现差异**：不同格式对同一算子的实现可能不同（如卷积的 padding 计算）
2. **量化损失**：从 FP32 转到 INT8/INT4 会带来精度损失（通常 < 1%）
3. **Opset 版本差异**：不同 ONNX Opset 版本对算子的定义可能不同
4. **自定义算子丢失**：部分自定义算子可能无法转换，需要回退或替换

**精度损失检测方法**：

- **数值对比**：转换前后对同一输入的输出进行数值对比
- **精度指标**：计算精度损失指标（如 MSE、MAE、余弦相似度）
- **实际测试**：在真实数据上测试转换后模型的精度

**精度损失缓解策略**：

- **选择合适的量化方式**：感知量化（Perceptual Quantization）可以保留关键层的精度
- **校准数据**：使用真实数据进行校准，提升量化后的精度
- **混合精度**：关键层使用 FP16/FP32，其他层使用 INT8
- **算子替换**：将精度敏感的算子替换为更高精度的实现

### 转换失败排查

模型转换失败是常见问题，常见原因和排查方法：

**算子不支持**
- 原因：目标格式不支持模型中的某个算子
- 排查：查看转换工具的错误日志，定位不支持的算子
- 解决：算子替换（如用 LayerNorm 替换自定义归一化）、自定义算子、回退到 CPU

**维度不匹配**
- 原因：输入输出维度不符合目标格式要求
- 排查：检查模型的输入输出形状（如 batch_size、sequence_length）
- 解决：设置动态维度（如 `-1`）、调整输入输出形状

**版本不兼容**
- 原因：ONNX Opset 版本、TFLite 版本、CoreML 版本不兼容
- 排查：检查版本号，查看官方文档
- 解决：升级或降级版本、调整算子使用方式

**内存不足**
- 原因：转换工具在转换过程中内存不足（常见于大模型）
- 排查：检查系统内存使用情况
- 解决：减小模型、分批转换、增加系统内存

## 格式间的兼容性权衡

### 框架支持度对比

不同格式对不同训练框架的支持度差异：

| 格式 | PyTorch | TensorFlow | Keras | MXNet | 自定义框架 |
|------|----------|-------------|--------|--------|-----------|
| ONNX | ★★★★★ | ★★★★★ | ★★★★☆ | ★★★☆☆ | ★★☆☆☆ |
| TFLite | ★★★★☆ | ★★★★★ | ★★★★☆ | ★★☆☆☆ | ★☆☆☆☆ |
| CoreML | ★★★★☆ | ★★★★☆ | ★★★☆☆ | ★★☆☆☆ | ★☆☆☆☆ |
| MNN | ★★★☆☆ | ★★★☆☆ | ★★★☆☆ | ★★☆☆☆ | ★☆☆☆☆ |
| NCNN | ★★★☆☆ | ★★★☆☆ | ★★★☆☆ | ★★☆☆☆ | ★☆☆☆☆ |
| GGUF | ★★★★☆（LLaMA）| ★☆☆☆☆ | ★☆☆☆☆ | ★☆☆☆☆ | ★☆☆☆☆ |

**关键洞察**：

1. **ONNX 的框架支持度最高**：几乎所有主流框架都支持导出到 ONNX
2. **TFLite 对 TensorFlow 支持度最高**：毕竟是 Google 的官方格式
3. **CoreML 对 PyTorch 支持度较高**：coremltools 对 PyTorch 的支持比 TensorFlow 更好
4. **自定义框架支持度低**：如果使用自定义训练框架，需要自己开发转换工具

### 硬件加速器支持度

不同格式对不同硬件加速器的支持度差异：

| 格式 | CPU | GPU | NPU | DSP |
|------|-----|-----|-----|-----|
| ONNX | ★★★★★ | ★★★★☆ | ★★★☆☆ | ★★☆☆☆ |
| TFLite | ★★★★★ | ★★★★★ | ★★★★★ | ★★★★☆ |

**关键洞察**：

1. **TFLite 和 CoreML 的硬件加速支持度最高**：毕竟是 Android 和 iOS 的官方格式
2. **ONNX 的硬件加速取决于运行时**：ONNX Runtime、TFLite、CoreML 都支持 ONNX，但硬件加速能力不同
3. **MNN/NCNN 对 GPU 和 NPU 的支持度较好**：专门针对移动端优化
4. **GGUF 主要依赖 CPU**：虽然支持 GPU，但端侧 LLM 推理主要依赖 CPU 优化

4. **GGUF 的学习成本中等**：需要了解 llama.cpp 生态，但文档丰富

### 格式选型决策树

端侧推理的格式选型需要综合考虑多个维度：

```
是否跨平台？
├─ 是 → 使用 ONNX 作为中间层
│        ├─ 是否需要支持 iOS？
│        │      ├─ 是 → ONNX → CoreML
│        │      └─ 否 → ONNX → TFLite
│        └─ 是否需要支持国产手机？
│               ├─ 是 → ONNX → MNN/NCNN
│               └─ 否 → ONNX → TFLite
└─ 否 → 只支持单一平台
         ├─ 只支持 Android → 直接用 TFLite
         ├─ 只支持 iOS → 直接用 CoreML
         ├─ 只支持国产手机 → 直接用 MNN/NCNN
         └─ 是否 LLM 场景？
                ├─ 是 → GGUF + llama.cpp
                └─ 否 → 根据平台选择 TFLite/CoreML/MNN/NCNN
```

**工程启示**：

1. **跨平台项目优先选 ONNX**：作为中间层，简化后续转换流程
2. **单一平台优先选官方格式**：降低转换成本，获得官方支持
3. **LLM 场景优先选 GGUF**：针对 LLM 优化，支持多种量化格式和超大模型
4. **国产手机优先选 MNN/NCNN**：针对 ARM 架构深度优化，可能有更好的性能

## 总结

端侧推理工程师需要深入理解常见端侧模型格式、转换工具链和选型权衡：

- **ONNX 是中间层**：跨框架支持度高，适合作为转换的中间格式
- **TFLite/LiteRT 是 Android 端首选**：官方支持，硬件加速好，生态成熟
- **CoreML 是 iOS 端首选**：官方支持，硬件优化好，内存占用低
- **MNN/NCNN 是国产优化方案**：针对 ARM 架构优化，适合国产手机
- **GGUF 是端侧 LLM 专用方案**：针对 LLM 优化，支持多种量化格式和超大模型

在实际部署中，格式选型需综合考虑平台支持度、硬件加速器支持度、转换成本、维护成本和模型特性。模型转换是端侧推理的必经之路，需要掌握转换工具链、注意事项和精度损失检测方法。

**记住**：端侧推理的核心目标不是追求最通用的格式，而是在满足功能和性能的前提下，选择最适合目标平台的格式，最小化转换成本和维护成本。
