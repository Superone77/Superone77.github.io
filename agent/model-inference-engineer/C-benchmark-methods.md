# 附录 C：性能基准测试方法

本附录介绍端侧大模型推理的性能基准测试方法，包括性能指标、基准测试工具、测试方法与流程、结果分析与优化。

---

## C.1 端侧推理性能指标

### C.1.1 延迟指标

**首字延迟（TTFT，Time To First Token）**
- 定义：从用户输入到生成第一个 token 所需的时间
- 测量单位：毫秒（ms）
- 典型值：10-500ms
- 测量方法：在用户输入和第一个 token 输出之间记录时间戳

**生成延迟（Generation Latency）**
- 定义：生成每个 token 所需的时间
- 测量单位：毫秒/token（ms/token）
- 典型值：5-50ms/token
- 测量方法：记录每个 token 的生成时间，计算平均值

**端到端延迟（End-to-End Latency）**
- 定义：从用户输入到完整输出所需的时间
- 测量单位：毫秒（ms）
- 典型值：50-2000ms
- 测量方法：在用户输入和完整输出之间记录时间戳

**冷启动延迟（Cold Start Latency）**
- 定义：模型首次加载到内存并完成第一次推理所需的时间
- 测量单位：毫秒（ms）
- 典型值：100-2000ms
- 测量方法：记录模型加载时间和首次推理时间

**热启动延迟（Warm Start Latency）**
- 定义：模型已在内存中，执行推理所需的时间
- 测量单位：毫秒（ms）
- 典型值：10-500ms
- 测量方法：在模型加载后，记录推理时间

### C.1.2 吞吐量指标

**Tokens Per Second（TPS）**
- 定义：每秒生成的 token 数
- 测量单位：tokens/秒
- 典型值：10-100 tokens/秒
- 测量方法：记录生成的总 token 数和总时间，计算 TPS

**Queries Per Second（QPS）**
- 定义：每秒处理的请求数
- 测量单位：requests/秒
- 典型值：1-100 requests/秒
- 测量方法：记录处理的总请求数和总时间，计算 QPS

**Batch Size**
- 定义：每次推理处理的样本数
- 测量单位：样本数/批次
- 典型值：1-32
- 测量方法：根据应用场景选择合适的 batch size

### C.1.3 内存指标

**模型内存占用（Model Memory Usage）**
- 定义：模型加载到内存后占用的内存大小
- 测量单位：MB 或 GB
- 典型值：100MB - 16GB
- 测量方法：使用性能分析工具测量模型加载后的内存占用

**KV Cache 内存占用（KV Cache Memory Usage）**
- 定义：KV Cache 占用的内存大小
- 测量单位：MB 或 GB
- 典型值：10MB - 1GB
- 测量方法：记录序列长度、层数、头数、头维度，计算 KV Cache 内存占用

**峰值内存占用（Peak Memory Usage）**
- 定义：推理过程中内存占用的最大值
- 测量单位：MB 或 GB
- 典型值：200MB - 32GB
- 测量方法：使用性能分析工具记录推理过程中的内存占用峰值

**内存碎片率（Memory Fragmentation Rate）**
- 定义：内存碎片占总内存的比例
- 测量单位：百分比（%）
- 典型值：5% - 20%
- 测量方法：使用性能分析工具分析内存碎片

### C.1.4 功耗指标

**功耗（Power Consumption）**
- 定义：推理过程中消耗的电能
- 测量单位：瓦（W）
- 典型值：1-10W
- 测量方法：使用功耗测量工具（如 Battery Historian）测量功耗

**能效比（Energy Efficiency）**
- 定义：每生成一个 token 消耗的电能
- 测量单位：毫焦/token（mJ/token）
- 典型值：1-10 mJ/token
- 测量方法：记录总功耗和总 token 数，计算能效比

**电池续航（Battery Life）**
- 定义：在电池供电下能持续推理的时间
- 测量单位：小时（h）
- 典型值：2-10h
- 测量方法：在满电状态下持续推理，记录电池耗尽的时间

### C.1.5 准确性指标

**Top-1 Accuracy**
- 定义：预测结果中最高概率的类别正确的比例
- 测量单位：百分比（%）
- 典型值：70% - 95%
- 测量方法：在测试集上计算 Top-1 Accuracy

**Top-5 Accuracy**
- 定义：预测结果中前 5 个最高概率的类别包含正确类别的比例
- 测量单位：百分比（%）
- 典型值：85% - 99%
- 测量方法：在测试集上计算 Top-5 Accuracy

**F1 Score**
- 定义：精确率和召回率的调和平均数
- 测量单位：百分比（%）
- 典型值：70% - 95%
- 测量方法：在测试集上计算 F1 Score

**Perplexity**
- 定义：模型对测试数据的拟合程度，越低越好
- 测量单位：无
- 典型值：10-100
- 测量方法：在测试集上计算 Perplexity

## C.3 测试方法与流程

### C.3.1 测试环境准备

**硬件环境**
- 设备选择：选择具有代表性的设备（如 iPhone 15 Pro、Samsung Galaxy S24）
- 设备数量：至少 3 台不同设备，覆盖高、中、低端
- 设备状态：关闭后台应用，关闭节能模式，连接电源
- 环境温度：25±5℃（避免高温影响性能）

**软件环境**
- 操作系统：最新稳定版本
- 运行时：测试目标运行时（如 TFLite、ONNX Runtime、CoreML）
- 模型版本：使用生产环境的模型版本
- 测试工具：选择合适的性能分析工具

**测试数据**
- 数据规模：至少 1000 个样本
- 数据分布：覆盖真实使用场景
- 数据格式：与生产环境一致

### C.3.2 测试方法

**冷启动测试**
1. 关闭应用
2. 清除后台应用
3. 启动应用
4. 加载模型
5. 执行推理
6. 记录冷启动延迟

**热启动测试**
1. 启动应用
2. 加载模型
3. 执行多次推理（如 100 次）
4. 记录每次推理延迟
5. 计算平均热启动延迟

**吞吐量测试**
1. 启动应用
2. 加载模型
3. 持续执行推理（如 1000 次）
4. 记录总时间和总请求数
5. 计算 QPS 和 TPS

**内存占用测试**
1. 启动应用
2. 加载模型
3. 使用性能分析工具记录内存占用
4. 执行推理
5. 记录峰值内存占用

**功耗测试**
1. 启动应用
2. 加载模型
3. 满电状态下持续推理（如 1 小时）
4. 记录总功耗和总 token 数
5. 计算能效比

### C.3.3 测试流程

**1. 制定测试计划**
- 明确测试目标：性能、内存、功耗
- 确定测试设备：高、中、低端设备
- 选择测试工具：Android Profiler、Xcode Instruments 等
- 准备测试数据：覆盖真实使用场景

**2. 搭建测试环境**
- 配置硬件环境：设备、温度、电源
- 配置软件环境：操作系统、运行时、测试工具
- 准备测试数据：数据规模、数据分布

**3. 执行测试**
- 按照测试方法执行测试
- 记录测试结果：延迟、吞吐量、内存、功耗
- 重复测试：至少 3 次，计算平均值和标准差

**4. 分析测试结果**
- 计算性能指标：TTFT、生成延迟、QPS、TPS
- 计算内存指标：模型内存占用、峰值内存占用
- 计算功耗指标：功耗、能效比
- 对比不同设备、不同配置的性能差异

**5. 输出测试报告**
- 汇总测试结果：表格、图表
- 分析性能瓶颈：延迟、吞吐量、内存、功耗
- 提出优化建议：量化、剪枝、蒸馏、硬件后端选择

## C.4 结果分析与优化

### C.4.1 性能瓶颈定位

**延迟瓶颈**
- 检查 TTFT：如果 TTFT 过长，可能是模型加载慢或 Prefill 阶段慢
- 检查生成延迟：如果生成延迟过长，可能是 Decode 阶段慢
- 检查网络延迟：如果是端云协同，可能是网络延迟过长

**吞吐量瓶颈**
- 检查 batch size：如果 batch size 过小，吞吐量可能较低
- 检查并发能力：如果单线程，吞吐量可能较低
- 检查硬件利用率：如果 NPU 利用率低，吞吐量可能较低

**内存瓶颈**
- 检查模型大小：如果模型过大，可能导致 OOM
- 检查 KV Cache：如果 KV Cache 过大，可能导致内存不足
- 检查内存碎片：如果内存碎片严重，可能导致内存不足

**功耗瓶颈**
- 检查硬件利用率：如果 GPU 利用率高，功耗可能较高
- 检查推理频率：如果推理频率高，功耗可能较高
- 检查后台活动：如果后台活动多，功耗可能较高

### C.4.2 优化策略

**延迟优化**
- 量化：使用 INT8 量化，减少计算量
- KV Cache 优化：减少 KV Cache 大小，使用量化 KV Cache
- 硬件后端选择：优先使用 NPU，其次是 GPU
- 模型压缩：使用剪枝、蒸馏等技术，减少模型大小

**吞吐量优化**
- 增加 batch size：如果内存允许，增加 batch size 提升吞吐量
- 多线程/多进程：利用多核 CPU 提升并发能力
- 模型并行：如果设备有多个 NPU/GPU，使用模型并行

**内存优化**
- 量化：使用 INT8 量化，减少模型大小
- KV Cache 优化：减少 KV Cache 大小，使用量化 KV Cache
- 模型分层：将模型分层，分时加载和推理
- 内存池：使用内存池，减少内存碎片

**功耗优化**
- 硬件后端选择：优先使用 NPU，能效比更高
- 降低频率：如果延迟允许，降低硬件频率
- 优化调度：减少后台活动，优化推理调度

### C.4.3 A/B 测试

**A/B 测试流程**
1. 设计 A/B 测试方案：明确测试目标、测试指标、测试时长
2. 选择测试设备：覆盖不同设备类型
3. 分配测试用户：A 组使用原模型，B 组使用优化模型
4. 收集测试数据：延迟、吞吐量、内存、功耗、用户体验
5. 分析测试结果：对比 A/B 组性能差异
6. 决策：如果 B 组性能更好，全量部署优化模型

**A/B 测试指标**
- 性能指标：TTFT、生成延迟、QPS、TPS
- 内存指标：模型内存占用、峰值内存占用
- 功耗指标：功耗、能效比
- 用户体验指标：用户满意度、任务完成率

## C.5 测试案例

### C.5.1 案例：iPhone 15 Pro 上的 LLM 推理性能测试

**测试环境**
- 设备：iPhone 15 Pro（A17 Pro，8GB RAM）
- 操作系统：iOS 17.2
- 运行时：CoreML
- 模型：7B 参数，INT8 量化
- 测试数据：1000 个对话样本

**测试结果**

| 指标 | 测试结果 |
|------|---------|
| **冷启动延迟** | 150ms |
| **热启动延迟** | 30ms |
| **生成延迟** | 10ms/token |
| **TPS** | 100 tokens/秒 |
| **模型内存占用** | 7GB |
| **峰值内存占用** | 7.5GB |
| **功耗** | 5W |
| **能效比** | 0.5 mJ/token |

**结论**
- 在 iPhone 15 Pro 上，7B 参数的 INT8 量化模型可以流畅运行
- TTFT 为 150ms，生成延迟为 10ms/token，用户体验良好
- 内存占用为 7GB，在设备内存限制内
- 功耗为 5W，能效比为 0.5 mJ/token，功耗可接受
