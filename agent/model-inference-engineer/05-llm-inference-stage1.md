# 端侧 LLM 推理链路：分词、提示词、预填充与解码、采样策略

端侧 LLM 推理的核心挑战在于**内存与计算资源的平衡**：分词（Tokenizer）占用、提示词（Prompt）长度、KV Cache 内存占用、预填充（Prefill）阶段的首字延迟（TTFT）、解码（Decode）阶段的逐 token 生成时间，都需要在有限资源约束下优化。

---

## Tokenizer 与文本编码

### Tokenizer 的作用与原理

Tokenizer 是 LLM 的**第一道门**：将原始文本转换为 token 序列，模型才能处理。Tokenizer 的质量直接影响：

- **推理延迟**：分词速度影响首字延迟（TTFT）
- **内存占用**：token 序列长度影响 KV Cache 内存占用
- **准确性**：分词粒度影响模型输出质量

端侧推理的 Tokenizer 优化与云侧不同：

- **性能优先**：端侧设备性能有限，需要高效的分词实现
- **内存敏感**：端侧设备内存有限，需要尽量减少 tokenizer 内存占用
- **延迟敏感**：端侧推理对响应延迟要求高，tokenizer 延迟不能成为瓶颈

### 常见 Tokenizer 类型

**BPE（Byte Pair Encoding）**
- **原理**：迭代合并字节对，形成最终的词汇表
- **优点**：压缩率高，词汇表紧凑，适合英文等语言
- **缺点**：不适合中文、日文等语言（无显式分词边界）
- **代表模型**：GPT 系列、LLaMA 系列

**Unigram**
- **原理**：基于字节级别建立词汇表，覆盖所有 Unicode 字符
- **优点**：覆盖所有字符，无未知 token
- **缺点**：词汇表大（256 字节 + 特殊标记），embedding 矩阵大
- **代表模型**：GPT-2、GPT-3

**SentencePiece**
- **原理**：结合 BPE 和 Unigram，基于语言模型的损失优化
- **优点**：平衡压缩率和语义粒度，适合多语言
- **缺点**：分词过程复杂
- **代表模型**：T5 系列、BERT 系列（部分）

**WordPiece**
- **原理**：与 SentencePiece 类似，但优化方式不同
- **优点**：支持多语言，平衡压缩率和语义粒度
- **缺点**：分词实现复杂
- **代表模型**：BERT 系列（部分）

### 端侧 Tokenizer 优化

端侧推理的 Tokenizer 优化策略：

**1. 量化与压缩**
- **量化 tokenizer 权重**：将 FP16 tokenizer 转为 INT8，减少内存占用
- **量化 embedding 矩阵**：使用 4-bit/8-bit 量化，减少 embedding 内存
- **压缩词汇表**：使用压缩算法（如哈希压缩）减少词汇表内存占用

**2. 性能优化**
- **避免 Python 开销**：端侧推理通常用 C++/Rust/Swift 实现，避免 Python 开销
- **使用 SIMD 优化**：利用 AVX/NEON 等向量指令加速分词
- **多线程分词**：将文本分块，并行分词，减少延迟

**3. 内存优化**
- **延迟加载 tokenizer**：不要一次性加载全部 tokenizer 参数，按需加载
- **共享词汇表**：多个模型共享同一个 tokenizer，减少重复内存
- **内存映射（Memory-mapped）加载**：对于大 tokenizer，使用内存映射避免复制

**4. 精度优化**
- **感知量化（Perceptual Quantization）**：对敏感层（如 tokenizer 的 embedding 层）保留更高精度
- **混合精度**：embedding 矩阵使用 FP16/FP32，其他层使用 INT8
- **校准数据**：使用真实数据校准量化后的精度

## Prompt 组织与拼接策略

### 系统提示的设计与作用

系统提示（System Prompt）是 LLM 的**行为控制器**，定义模型的身份、任务风格、约束条件。端侧推理的系统提示优化需要考虑：

- **端侧约束**：系统提示需要明确端侧约束（如计算资源限制、响应时间限制）
- **隐私保护**：系统提示需要明确隐私保护规则（如不生成敏感信息）
- **个性化配置**：系统提示可以包含用户个性化配置（如语言风格、输出格式）

端侧推理的系统提示特点：

**1. 静态 vs 动态系统提示**
- **静态系统提示**：预定义的系统提示，硬编码到应用中，不可修改
- **动态系统提示**：通过配置文件或远程更新系统提示，支持 A/B 测试和灰度发布

**2. 系统提示压缩**
- **关键词压缩**：使用关键词缩写（如 "你是一个专业的代码助手" → "专业代码助手"）
- **模板化系统提示**：使用模板化系统提示，支持动态变量替换
- **系统提示蒸馏**：将长系统提示蒸馏为短系统提示，减少推理开销

### 用户输入的处理与编码

用户输入的处理与编码是端侧推理的**关键瓶颈**：用户输入需要经过预处理、分词、编码，才能输入模型。

端侧推理的用户输入处理策略：

**1. 输入预处理**
- **文本清洗**：去除无效字符、统一换行符、去除多余空格
- **长度限制**：根据模型的最大上下文长度限制输入，避免 OOM
- **格式标准化**：统一输入格式（如 Markdown、纯文本、JSON）

**2. 分词优化**
- **批量分词**：如果用户输入很长，分块分词，避免阻塞 UI 线程
- **缓存分词结果**：对重复输入（如多轮对话中的相同输入），缓存分词结果
- **延迟加载 tokenizer**：用户输入到达前，提前预热 tokenizer

**3. 编码优化**
- **使用高效的编码库**：如 SentencePiece 的 C++ 实现，避免 Python 开销
- **向量化编码**：使用 SIMD 优化编码过程
- **异步编码**：对于长输入，异步编码，避免阻塞 UI

### 历史消息的组织与截断策略

多轮对话需要管理历史消息，而端侧设备的内存有限，因此需要**截断策略**。

端侧推理的历史消息组织策略：

**1. 消息格式转换**
- **将消息列表转换为对话模板**：将 `["user": A, "assistant": B, "user": C]` 转换为对话模板格式（如 `<|user|>\n<|assistant|>\n<|user|>\n<|assistant|>`）
- **添加系统提示前缀**：在消息列表前添加系统提示
- **特殊处理系统消息**：系统消息（如工具调用结果）不加入历史消息，而是直接拼接

**2. 截断策略**
- **基于 token 数量截断**：根据模型的最大上下文长度（如 4096 tokens），从最新消息开始倒序截断，保留最近 N 条消息
- **基于优先级截断**：系统提示 > 最近用户输入 > 历史消息，优先保留系统提示和最近用户输入
- **滑动窗口**：使用滑动窗口截断，避免完全丢失早期对话上下文

**3. 内存优化**
- **延迟加载历史消息**：不要一次性加载全部历史消息，按需加载
- **压缩历史消息**：对历史消息进行压缩（如去除重复内容、缩短系统提示）
- **共享 KV Cache**：多轮对话共享同一个 KV Cache，避免重复计算

### 模板化 Prompt 的实现

模板化 Prompt 可以**提高开发效率**和**支持动态配置**。端侧推理的模板化 Prompt 实现策略：

**1. 模板化系统提示**
- **使用模板变量**：如 `{{model_name}}`、`{{language}}`、`{{constraints}}`，在运行时替换
- **支持多语言**：不同语言使用不同的系统提示模板
- **支持个性化**：根据用户偏好动态调整系统提示（如风格、语气）

**2. 模板化对话格式**
- **使用模板变量**：如 `{{history}}`、`{{user_input}}`、`{{context}}`，在运行时拼接
- **支持多种对话格式**：如 ChatML、LLaMA 格式、Mistral 格式
- **支持特殊指令**：如工具调用、代码生成、文本生成的特殊指令

**3. 模板化实现**
- **使用高效的模板引擎**：如 Jinja2 的 C++ 端口、Mustache 的 Rust 实现，避免字符串拼接开销
- **预编译模板**：在应用启动时预编译模板，避免运行时解析开销
- **缓存模板结果**：对重复使用的模板，缓存拼接结果

## Prefill 与 Decode 阶段

### Prefill 阶段的特点与作用

Prefill 阶段是 LLM 推理的**第一阶段**：处理历史上下文（包括系统提示、历史消息、当前用户输入），生成完整的 KV Cache。Prefill 阶段的特点：

- **计算密集型**：需要对输入文本进行大规模的矩阵乘法（MatMul）
- **内存密集型**：需要生成完整的 KV Cache，内存占用大
- **可并行化**：Prefill 阶段可以并行处理多个层，提升计算效率
- **一次性输出**：Prefill 阶段一次性生成所有 token 的 KV Cache，没有逐 token 生成

端侧推理的 Prefill 阶段优化策略：

**1. 计算优化**
- **使用专用硬件**：优先使用 NPU/GPU 进行 Prefill 阶段，提升计算效率
- **批量处理**：如果内存允许，批量处理多个输入，提升吞吐量
- **算子融合**：将 Prefill 阶段的多个算子融合为一个算子，减少内存访问

**2. 内存优化**
- **KV Cache 量化**：将 KV Cache 量化为 INT8，减少内存占用
- **KV Cache 压缩**：使用压缩算法（如稀疏化、低位宽）减少 KV Cache 内存
- **分片加载**：将模型分片加载，只加载 Prefill 阶段需要的层，减少内存占用

**3. 延迟优化**
- **异步 Prefill**：如果 UI 线程不需要立即响应，可以异步 Prefill，减少用户感知延迟
- **流水线并行**：将 Prefill 阶段与用户输入处理并行，减少整体延迟
- **热点数据缓存**：缓存常用的 KV Cache（如系统提示的 KV Cache），避免重复计算

### 首字延迟（TTFT）影响因素

首字延迟（Time To First Token，TTFT）是 LLM 推理的**关键指标**，从用户输入到第一个 token 输出的时间。TTFT 的影响因素：

**1. Prefill 阶段延迟**
- **输入长度**：输入越长，Prefill 阶段时间越长，TTFT 越大
- **模型层数**：模型层数越多，Prefill 阶段时间越长，TTFT 越大
- **硬件算力**：硬件算力越强，Prefill 阶段越快，TTFT 越小

**2. 系统提示长度**
- **系统提示越短**：系统提示越短，Prefill 阶段时间越短，TTFT 越小
- **系统提示越长**：系统提示越长，Prefill 阶段时间越长，TTFT 越大
- **系统提示优化**：使用压缩系统提示、蒸馏系统提示，减少系统提示长度

**3. 历史消息数量**
- **历史消息越少**：历史消息越少，Prefill 阶段时间越短，TTFT 越小
- **历史消息越多**：历史消息越多，Prefill 阶段时间越长，TTFT 越大
- **截断策略**：从最新消息开始截断，减少历史消息数量，缩短 TTFT

**4. 硬件与软件优化**
- **硬件选择**：使用 NPU/GPU 进行 Prefill 阶段，提升计算效率
- **软件优化**：使用高效的 tokenizer、异步 Prefill、流水线并行，减少 TTFT

### Decode 阶段的逐 token 生成

Decode 阶段是 LLM 推理的**第二阶段**：基于 Prefill 阶段生成的 KV Cache，逐 token 生成输出。Decode 阶段的特点：

- **顺序依赖型**：每个 token 的生成依赖于前一个 token 的 KV Cache，不能并行
- **计算轻量型**：每个 token 的生成只需要少量计算（如采样、归一化、输出投影）
- **内存低占用型**：Decode 阶段只需要存储当前的 KV Cache，内存占用增长缓慢

端侧推理的 Decode 阶段优化策略：

**1. 计算优化**
- **使用高效采样算法**：如 nucleus sampling、top-k sampling，减少采样计算开销
- **向量化采样**：使用 SIMD 优化采样计算，提升采样效率
- **缓存 KV Cache 访问**：缓存 KV Cache 的访问模式，减少内存访问开销

**2. 内存优化**
- **KV Cache 量化**：将 Decode 阶段新生成的 KV Cache 量化为 INT8，减少内存占用
- **KV Cache 复用**：多个轮对话共享同一个 KV Cache 的 Prefill 阶段部分，避免重复计算
- **延迟释放 KV Cache**：对话结束后，及时释放 KV Cache，释放内存

**3. 延迟优化**
- **流式输出（Streaming）**：在 Decode 阶段使用流式输出，每生成一个 token 就输出一次，减少用户感知延迟
- **预测性解码**：使用预测性解码（Speculative Decoding），用小模型预测大模型输出，减少 Decode 阶段延迟
- **热点缓存**：缓存常用的 KV Cache（如系统提示的 KV Cache、常用词的 KV Cache），减少重复计算

### 两个阶段的性能优化策略

端侧推理的两个阶段（Prefill + Decode）需要协同优化：

**1. 硬件分配**
- **NPU/GPU 进行 Prefill**：Prefill 阶段计算密集，使用 NPU/GPU 提升计算效率
- **CPU/NPU 进行 Decode**：Decode 阶段顺序依赖，使用 CPU/NPU 提升能效比
- **避免硬件冲突**：避免 Prefill 阶段和 Decode 阶段同时抢占 GPU/NPU，导致性能下降

**2. 内存管理**
- **KV Cache 池化**：使用对象池复用 KV Cache 内存，减少内存分配/释放开销
- **KV Cache 分片**：将 KV Cache 分片存储，按需加载，减少单次内存占用
- **KV Cache 压缩**：使用压缩算法（如 INT8 量化、稀疏化）减少 KV Cache 内存占用

**3. 流水线并行**
- **Prefill + Decode 流水线**：如果内存允许，可以同时进行 Prefill 和 Decode（如并行处理多个输入），提升吞吐量
- **异步处理**：Prefill 阶段和 Decode 阶段异步处理，减少整体延迟
- **双缓冲机制**：使用双缓冲机制，一个缓冲区进行 Prefill，另一个缓冲区进行 Decode，减少等待时间

## Sampling 策略与参数

### 贪婪采样与温度

**贪婪采样（Greedy Sampling）**：选择概率最高的 token，不引入随机性。
- **优点**：输出稳定，重复性低，适合任务型场景（如代码生成、文本分类）
- **缺点**：输出缺乏多样性，容易陷入循环（如重复生成相同的短语）
- **端侧适用性**：适合端侧推理的低功耗场景，因为计算开销小

**温度（Temperature）**：控制输出的随机性，温度越高，输出越随机。
- **温度 = 0**：确定性输出，等价于贪婪采样
- **温度 = 0.7**：平衡随机性和稳定性，适合对话场景
- **温度 = 1.0+**：高随机性，适合创意场景
- **端侧适用性**：温度参数需要根据场景调整，计算开销小

### top-k 与 top-p 采样

**top-k 采样**：从概率最高的前 k 个 token 中随机选择一个。
- **优点**：输出稳定性高，适合任务型场景
- **缺点**：k 值选择不当会导致输出单一或错误
- **端侧适用性**：适合端侧推理的低功耗场景，因为计算开销小

**top-p 采样（Nucleus Sampling）**：从概率最高的前 p 个 token 中随机选择一个，直到累计概率超过 p。
- **优点**：输出多样性好，稳定性高，适合对话场景
- **缺点**：p 值选择不当会导致输出不稳定
- **端侧适用性**：适合端侧推理的对话场景，因为计算开销适中

### nucleus sampling

**nucleus sampling**：动态调整 top-p，选择累积概率超过阈值的最小 token 集合。
- **优点**：输出多样性好，稳定性高，适合对话场景
- **缺点**：阈值选择不当会导致输出不稳定
- **端侧适用性**：适合端侧推理的对话场景，因为计算开销适中

### 采样参数对输出质量的影响

采样参数对端侧推理的输出质量影响显著，需要根据场景调整：

**1. 任务型场景**（如代码生成、文本分类）
- **参数推荐**：温度 = 0、top-k = 1 或 top-p = 0.9（接近贪婪采样）
- **输出质量**：稳定性高，重复性低，准确率高
- **计算开销**：计算开销小，适合端侧推理的低功耗场景

**2. 对话场景**（如聊天机器人、智能助手）
- **参数推荐**：温度 = 0.7、top-k = 40 或 top-p = 0.9
- **输出质量**：多样性好，稳定性高，适合多轮对话
- **计算开销**：计算开销适中，适合端侧推理的中功耗场景

**3. 创意场景**（如创意写作、诗歌生成）
- **参数推荐**：温度 = 1.0+、top-k = 40 或 top-p = 0.95
- **输出质量**：多样性好，创意性强，适合创意场景
- **计算开销**：计算开销大，适合端侧推理的高功耗场景

端侧推理的采样参数优化策略：

**1. 动态调整参数**
- **根据场景调整参数**：不同场景使用不同的温度、top-k、top-p 参数
- **根据用户偏好调整参数**：支持用户自定义采样参数，满足个性化需求
- **A/B 测试**：通过 A/B 测试找到最优采样参数，提升输出质量

**2. 性能优化**
- **使用高效的采样算法**：如 nucleus sampling 的优化实现，减少采样计算开销
- **向量化采样**：使用 SIMD 优化采样计算，提升采样效率
- **缓存采样结果**：缓存常用的采样分布（如高频 token 的概率分布），减少采样计算开销

---

## 总结

端侧 LLM 推理链路的优化需要综合考虑 tokenizer、提示词、prefill/decode、采样 等多个阶段：

- **Tokenizer 优化**：量化与压缩、性能优化、内存优化、精度优化
- **Prompt 组织优化**：静态 vs 动态系统提示、输入预处理、分词优化、编码优化、历史消息组织与截断、模板化实现
- **Prefill/Decode 优化**：计算优化、内存优化、延迟优化、两个阶段协同优化
- **Sampling 优化**：动态调整参数、性能优化、A/B 测试

在实际部署中，端侧 LLM 推理的优化需要根据硬件能力、软件栈兼容性、用户偏好综合调整。推理框架通过优化的 tokenizer、提示词组织、prefill/decode、采样 实现，提供高效的端侧推理能力。

**记住**：端侧 LLM 推理的核心目标不是追求最快的 TTFT 或最高的吞吐量，而是在有限资源约束下实现可接受的性能和质量平衡。
