# 端侧硬件加速器总览

端侧推理的核心挑战在于**硬件资源约束**：功耗、散热、内存带宽、算力利用率都在有限空间内争夺。理解 NPU、GPU、CPU 的能力边界与调度策略，是端侧推理工程师的基础。

---

## NPU：面向神经网络推理的专用处理器

### 设计理念

NPU（Neural Processing Unit）是专门为神经网络推理设计的加速器。与通用处理器不同，NPU 通过**专用乘加单元**、**高速片上内存**和**并行架构**实现高并行与低能耗，尤其适合矩阵乘法密集的推理任务。

NPU 的核心优化点在于：

- **数据流优化**：设计数据移动路径，减少片外内存访问，提升能效。典型设计是采用 systolic array（脉动阵列）或专用数据通路，让数据在计算单元间流动而非反复加载卸载。
- **算子加速**：针对 Conv、MatMul、Attention 等常见算子提供硬件支持。这些算子在 CNN 和 Transformer 中占比极高，硬件加速可以带来 10-100 倍的能效提升。
- **多精度支持**：INT8、INT4、FP16 等低精度计算，减少内存占用和功耗。INT8 量化后模型大小减半，内存带宽需求减半，而精度损失通常小于 1%。

NPU 的架构设计通常围绕**计算密度**和**能效比**而非峰值算力。例如，Apple Neural Engine（ANE）采用 16 核设计，每核 128 个 MAC 单元，峰值算力约 18 TOPS（INT8），但功耗仅 5W 左右；而同代手机 GPU 峰值算力可达 1+ TFLOPS（FP16），但功耗在 10-20W 范围。

### 在端侧推理中的定位

NPU 在端侧推理中扮演**主力加速器**的角色，尤其适合低功耗、低时延、空间受限的环境。典型场景包括：

- **语音识别与合成**：后台常驻 AI 需要持续监听或响应，功耗预算严格。NPU 的低功耗特性使其成为首选，例如 Apple Siri、Google Assistant 的端侧 ASR（自动语音识别）。
- **低分辨率视觉检测**：如人脸检测、手势识别、二维码扫描。这些任务计算量中等，但对响应延迟要求高，NPU 可以在 10-50ms 内完成。
- **端侧小模型推理**：如量化后的 7B、3B LLM。经过 INT4 量化后，模型参数大小可压缩到 4GB 以下，配合 NPU 的片上 SRAM 缓存，可以实现每秒 10-30 token 的生成速度。
- **实时视频处理**：如背景虚化、超分辨率、视频风格迁移。这些任务对吞吐量要求高，NPU 的并行架构可以充分利用视频帧之间的时空相关性。

NPU 的优势在于**能效比**明显优于 CPU 和 GPU。根据行业研究，在相同推理任务下，NPU 的功耗通常只有 GPU 的 1/3-1/5，是 CPU 的 1/2-1/3。对于电池驱动的移动设备，这意味着推理任务对续航的影响可以控制在可接受范围内。

### 端侧 NPU 的代表性平台

不同平台的 NPU 设计差异较大，端侧推理工程师需要了解其特性：

**Apple Neural Engine（ANE）**
- 集成在 A 系列芯片中（A11 及以上）
- 支持格式：CoreML 格式，通过 CoreML 框架调用
- 特点：统一内存架构，模型参数常驻内存，数据移动成本极低
- 约束：只支持特定算子集，自定义算子需要回退到 CPU/GPU
- 最佳实践：优先使用 ANE，回退到 Metal GPU（用于高算力需求），最后是 CPU

**高通 Hexagon DSP / NPU**
- 集成在骁龙芯片中（如骁龙 8 Gen 1、Gen 2）
- 支持格式：SNPE（Snapdragon Neural Processing Engine）格式、TFLite（通过 NNAPI）
- 特点：多代演进，从 DSP 逐步演进到专用 AI 加速器
- 约束：模型需要通过 SNPE 工具链转换，算子支持度依赖于驱动版本
- 最佳实践：使用 SNPE 工具链转换模型，优先选择 Hexagon NPU 后端

**三星 NPU**
- 集成在 Exynos 芯片中
- 支持格式： proprietary 格式、TFLite（通过 NNAPI）
- 特点：与 Android NNAPI 深度集成
- 约束：开发者文档相对较少，社区支持度较低
- 最佳实践：通过 NNAPI 统一抽象，避免直接调用厂商私有 API

**华为麒麟 NPU**
- 集成在 Kirin 芯片中
- 支持格式：DaVinci 架构专用格式
- 特点：专注 Transformer 优化，适合大模型推理
- 约束：生态相对封闭，主要服务于华为系产品
- 最佳实践：利用昇腾推理工具链，但需考虑跨平台兼容性

### 限制与约束

尽管 NPU 能效出色，但也有明显的工程约束：

- **算力天花板**：峰值算力通常低于同代 GPU，不适合超大模型。例如，端侧 NPU 峰值算力通常在 10-50 TOPS（INT8），而同代 GPU 可达 100+ TFLOPS（FP16）。对于未经极致压缩的 70B+ LLM，NPU 可能无法满足实时推理需求。
- **算子支持受限**：部分自定义算子可能无法直接映射到 NPU，需要回退到 CPU/GPU。例如，一些研究中的新型激活函数、自定义归一化层、稀疏化算子可能在 NPU 上无法高效执行。常见的解决方式包括：
  - **算子替换**：将自定义算子替换为 NPU 支持的等价算子（如用 LayerNorm 替换自定义归一化）
  - **算子融合**：将多个算子合并为一个 NPU 支持的 fused kernel
  - **CPU/GPU 回退**：将不支持的算子放到其他硬件上执行
- **软件栈依赖强**：不同厂商 NPU 的支持格式不同，需要适配特定工具链。这给跨平台部署带来挑战：
  - **Apple 生态**：必须使用 CoreML 格式，通过 Xcode 工具链转换
  - **Android 生态**：优先通过 NNAPI 统一抽象，但不同厂商的驱动质量差异大
  - **跨平台需求**：需要考虑 ONNX、TFLite 等中间格式，然后针对不同平台做二次转换
- **调试与可观测性差**：NPU 的内部执行细节通常不对外暴露，性能分析和问题排查难度高。相比之下，GPU 和 CPU 有成熟的 profiling 工具（如 Metal Performance HUD、perf）。

## GPU：通用并行计算加速器

### 端侧 GPU 的优势

GPU 通过**大量并行核心**提供高算力，适合矩阵乘法和卷积等并行计算任务。在端侧推理中，GPU 常作为**后备加速器**，用于 NPU 无法覆盖的场景：

- **较大模型的推理**：如未经极致压缩的 LLM（7B FP16、13B INT8）。这些模型计算量大，NPU 的算力可能不足以满足实时推理需求，而 GPU 的高吞吐量可以提供更好的性能。
- **高分辨率图像处理**：如超分辨率（4K→8K）、风格迁移、实时视频增强。这些任务对显存带宽要求高，GPU 的专用显存可以提供更高的数据吞吐量。
- **自定义算子较多的模型**：需要 GPU 通用性支持。例如，一些研究模型使用了非标准算子或动态图，NPU 无法映射，而 GPU 通过 OpenCL、Metal、CUDA 等通用编程接口可以实现灵活的 kernel 实现。

GPU 的峰值算力通常高于 NPU，在纯性能测试中往往更快。但代价是**高功耗**和**散热压力**，尤其在电池设备上会显著缩短续航。根据测试，在相同推理任务下，端侧 GPU 的功耗可能是 NPU 的 2-3 倍，这会导致设备温度升高、电池续航下降。

### 端侧 GPU 的代表性平台

**Apple GPU（Metal）**
- 集成在 A 系列芯片中，采用统一内存架构
- 支持框架：Metal Performance Shaders（MPS）、CoreML GPU 后端
- 特点：统一内存架构，CPU 和 GPU 共享内存，数据拷贝成本低
- 优势：Metal 编程模型成熟，社区支持度高，有丰富的 kernel 库
- 约束：GPU 与 NPU、CPU 共享散热空间，同时高负载会导致热降频
- 最佳实践：用于高算力需求任务，避免与 UI 渲染任务抢占 GPU 资源

**Android GPU（Adreno、Mali）**
- Adreno（高通骁龙系列）、Mali（三星 Exynos、联发科天玑）
- 支持框架：OpenCL、Vulkan、OpenGL ES
- 特点：不同厂商的 GPU 架构差异大，性能优化需要针对性调优
- 优势：通用性强，支持自定义 kernel 实现
- 约束：功耗高，散热压力大，电池设备上需谨慎使用
- 最佳实践：优先使用 TFLite GPU Delegate，通过 Vulkan 后端获得更好的性能

**跨平台 GPU（OpenCL、Vulkan）**
- 提供统一的 GPU 抽象层
- 适合跨平台推理框架（如 ONNX Runtime）
- 约束：不同 GPU 的性能差异大，需要针对不同硬件做优化

### GPU 推理的优化要点

端侧 GPU 推理的优化重点在于**算力利用率**和**内存带宽**：

**Kernel 融合**
- 将多个算子合并为一个 kernel，减少显存访问。例如，将 Conv + ReLU + BatchNorm 融合为一个 kernel，只需要一次读写显存，而不是三次。
- 融合的收益在显存带宽受限的场景下尤其明显。端侧 GPU 的显存带宽通常在 50-200 GB/s，远低于桌面 GPU（500+ GB/s），因此减少显存访问是关键。
- 实现方式：通过推理框架的算子融合机制（如 TFLite 的 OpResolver、ONNX Runtime 的 Graph Optimizer），或手动编写 fused kernel。

**动态调度**
- 根据设备负载动态分配 GPU 资源，避免抢占 UI 渲染。例如，当用户滑动屏幕、播放动画时，降低推理任务的 GPU 优先级。
- 调度策略通常由操作系统负责，但推理框架可以通过以下方式影响调度：
  - **设置 GPU 优先级**：Metal 中的 command queue priority、Vulkan 中的 queue priority
  - **分时复用**：将推理任务切分为多个小批次，在 UI 空闲时执行
  - **负载感知**：通过监听系统负载，动态调整推理任务的并发度

**内存对齐与批处理**
- 优化数据布局，提升内存访问局部性。例如，使用 NHWC（NCHW 优化为 NHWC，提升 GPU 缓存命中率）数据格式。
- 批处理可以提升 GPU 利用率，但会增加延迟。端侧推理通常 batch size = 1，因此需要针对单 batch 场景优化 kernel。
- 内存对齐：确保 tensor 的起始地址是 256 字节或 512 字节对齐，提升显存访问效率。

### GPU 推理的性能瓶颈

GPU 推理的主要瓶颈在于**功耗和散热**：

- **功耗**：GPU 满载时的功耗可达 10-20W，在电池设备上会导致续航显著下降。根据实测，持续运行 GPU 推理 1 小时可能消耗 20-40% 的电量。
- **散热**：GPU 高负载会导致设备温度升高，触发热降频。降频后性能可能下降 50-70%，抵消了 GPU 的性能优势。
- **电池寿命**：长期高功耗使用会加速电池老化，影响设备寿命。

因此，端侧推理工程师需要谨慎使用 GPU，优先考虑 NPU，只在必要时启用 GPU 加速。

## CPU：推理的兜底与控制平面

### CPU 推理的适用场景

CPU 是端侧推理的**兜底方案**，主要适用于：

- **轻量模型**：推理延迟要求不严、计算量小的任务。例如，简单的线性回归、小规模 CNN（如 MobileNet V1）。
- **算子回退**：NPU/GPU 不支持的算子需要在 CPU 上执行。这是常见的场景，因为 NPU 的算子支持度有限，GPU 的 kernel 也有约束，CPU 作为通用处理器可以执行任意算子。
- **控制逻辑**：模型编排、数据预处理、后处理等逻辑。例如：
  - 数据预处理：图像 resize、归一化、数据类型转换
  - 模型编排：根据输入选择不同的模型分支、动态调整模型参数
  - 后处理：模型输出的解析、阈值过滤、结果排序
- **单实例推理**：当只有少量并发请求时，CPU 的灵活性优势明显，避免了 GPU/NPU 的初始化开销。

CPU 的优势在于**通用性**和**可用性**：任何设备都有 CPU，软件栈成熟（SIMD、多核调度、各种推理框架均支持）。但在纯推理任务中，CPU 的能效比明显低于 NPU 和 GPU，通常只作为辅助方案。

### CPU 推理优化

CPU 推理的优化重点在于**指令集利用**和**多核调度**：

**SIMD 指令**
- AVX（x86）、NEON（ARM）等向量指令加速矩阵运算。通过单条指令同时处理多个数据，可以提升 2-4 倍性能。
- 实现方式：使用编译器 intrinsics、调用优化库（如 ARM Compute Library、Intel MKL-DNN），或由推理框架自动生成 SIMD 代码（如 ONNX Runtime 的 CPU Execution Provider）。
- 注意事项：SIMD 优化的效果依赖于数据对齐和内存访问模式，不合理的实现可能导致性能下降。

**多线程调度**
- 合理分配线程到不同核心，避免核心抢占和缓存失效。例如：
  - 将推理任务绑定到性能核心（而非能效核心），以获得更好的单线程性能
  - 控制并发线程数，避免超线程（SMT）导致的竞争
  - 使用 NUMA-aware 调度，在多 NUMA 节点服务器上优化内存访问
- 调度策略：通过线程池（如 OpenMP、TBB）、亲和性绑定（如 pthread_setaffinity_np）、操作系统调度接口（如 nice、ionice）实现。

**缓存友好**
- 优化数据访问模式，利用 L1/L2 缓存减少内存访问。例如：
  - 使用紧凑的数据格式（如 FP16、INT8），增加缓存命中率
  - 优化循环顺序，确保数据访问是连续的（cache line alignment）
  - 预取数据（prefetch），提前将需要的数据加载到缓存
- 实现方式：手动优化数据布局（如将 NCHW 转换为 NHWC），或使用推理框架的内存优化器。

### CPU 推理的性能瓶颈

CPU 推理的主要瓶颈在于**算力不足**和**内存带宽限制**：

- **算力**：CPU 核心数有限，峰值算力远低于 NPU/GPU。例如，现代手机 CPU 的峰值算力约为 50-200 GFLOPS（FP32），而同代 NPU 可达 10-50 TOPS（INT8），GPU 可达 100+ TFLOPS（FP16）。
- **内存带宽**：CPU 通过系统总线访问内存，带宽受限。端侧设备的内存带宽通常在 10-50 GB/s，而 NPU 的片上 SRAM 带宽可达 1+ TB/s，GPU 的显存带宽可达 50-200 GB/s。
- **并发能力**：多核调度有开销，且推理任务往往无法完美并行。例如，Transformer 的自回归生成是顺序依赖的，无法充分利用多核。

因此，CPU 推理通常只作为辅助方案，优先使用 NPU/GPU 加速。

## 异构计算调度

### 端侧异构计算架构

端侧设备通常配备**异构硬件**：CPU、GPU、NPU 可能同时存在。推理框架需要根据设备能力和模型需求，动态选择计算设备，实现**算力利用率最大化**和**功耗最小化**。

典型架构：

```
┌─────────────────────────────────────────┐
│           推理框架（Runtime）              │
│   • 模型编排与调度                        │
│   • Delegate 机制（硬件后端委托）         │
│   • 数据流管理与内存分配                  │
└─────────────────────────────────────────┘
            │
    ┌───────┼───────┬────────┐
    ▼       ▼       ▼        ▼
  NPU     GPU     CPU     其他（DSP/TPU）
```

在这个架构中，**推理框架**（如 CoreML、TFLite、ONNX Runtime）负责模型解析、图优化、资源管理，并通过 **Delegate** 机制将子图卸载到特定硬件后端。

### 动态计算设备选择

推理框架通常通过 **Delegate** 机制将子图卸载到特定硬件后端：

**NPU Delegate**
- 优先尝试 NPU，若失败则回退
- 检查每个算子是否在 NPU 的支持列表中
- 对于不支持的算子，有两种处理方式：
  - **图切分**：将不支持的算子从 NPU 子图中剥离，放到其他硬件上执行
  - **算子替换**：将不支持的算子替换为 NPU 支持的等价算子（可能带来精度损失）
- 实现示例：CoreML 中的 `computeUnits` 参数（`.cpuOnly`、`.cpuAndGPU`、`.all`）

**GPU Delegate**
- 适用于 NPU 不支持的算子或需要更高算力的子图
- 检查 GPU 的内存容量是否足够（避免 OOM）
- 检查 GPU 的负载是否过高（避免抢占 UI 渲染）
- 实现示例：TFLite 的 GPU Delegate（`TfLiteGpuDelegateV2Create`）、CoreML 的 GPU 后端

**CPU Fallback**
- 所有无法卸载到加速器的算子在 CPU 上执行
- CPU 的算力利用率通常较低，但可以作为兜底方案
- 实现示例：TFLite 的 CPU Backend、ONNX Runtime 的 CPU EP

**设备选择策略需考虑**：

- **算子支持度**：检查每个算子是否能在目标硬件上高效执行。例如，Attention 算子在 NPU 上可能不被支持，需要回退到 GPU/CPU。
- **数据移动成本**：频繁在不同设备间搬运数据会抵消加速收益。例如，如果 NPU 子图和 CPU 子图频繁切换，数据拷贝的开销可能超过 NPU 的加速收益。
- **功耗与散热**：优先选择能耗低的硬件，避免设备过热。例如，在电池低电量时，可以强制回退到 CPU，降低功耗。

### 数据流与任务调度

异构调度不仅是设备选择，还包括**数据流优化**：

**内存复用**
- 减少跨设备的数据拷贝，共享内存或 DMA。例如：
  - Apple 的统一内存架构：CPU、GPU、NPU 共享内存，数据拷贝成本极低
  - Android 的共享内存：通过 ION、DMABUF 实现跨设备内存共享
  - PCIe DMA：在服务器端，通过 DMA 引擎实现 GPU 和 CPU 之间的高效数据传输
- 实现方式：使用推理框架的内存管理器（如 TFLite 的 `TfLiteTensor` 内存池），或手动管理共享内存。

**流水线并行**
- 将模型分段到不同硬件，形成流水线并行。例如：
  - 第 1 层在 NPU 上执行，第 2 层在 GPU 上执行，第 3 层在 CPU 上执行
  - 优点：充分利用不同硬件的优势，提升整体吞吐量
  - 缺点：增加了调度复杂度，可能导致数据同步开销
- 实现方式：使用推理框架的子图切分机制（如 ONNX Runtime 的 SessionOptions），或手动切分模型。

**任务优先级**
- 确保推理任务不抢占关键 UI 渲染或系统服务。例如：
  - 在用户交互时，降低推理任务的优先级
  - 在设备空闲时，提升推理任务的优先级
- 实现方式：通过操作系统的调度接口（如 QoS Class、nice 值），或推理框架的优先级 API。

### 能耗优化策略

端侧推理的能耗优化是系统级工程，涉及：

**算力降级**
- 在电池低电量时自动降低算力或回退到 CPU。例如：
  - 当电量 < 20% 时，强制使用 CPU 推理，避免 GPU/NPU 的高功耗
  - 当电量 > 80% 时，优先使用 NPU/GPU，追求性能
- 实现方式：监听电池状态（如 iOS 的 UIDeviceBatteryState、Android 的 BatteryManager），动态调整推理策略。

**动态频率调节**
- 根据负载动态调整硬件频率，平衡性能与功耗。例如：
  - 在推理任务轻载时，降低 GPU/NPU 频率，减少功耗
  - 在推理任务重载时，提升 GPU/NPU 频率，保证性能
- 实现方式：通过硬件厂商的 API（如 Samsung DVFS、Qualcomm Turbo），或操作系统的电源管理接口。

**延迟感知调度**
- 在延迟允许范围内优先使用低功耗硬件。例如：
  - 对于实时性要求低的任务（如后台语音识别），使用 CPU 推理
  - 对于实时性要求高的任务（如实时视频分析），使用 NPU/GPU
- 实现方式：根据任务的 QoS 要求，选择不同的硬件后端。

## NPU vs GPU vs CPU：端侧推理能力对比

下表总结了三种硬件在端侧推理中的定位、优势与限制：

| 维度 | NPU | GPU | CPU |
|------|-----|-----|-----|
| **核心定位** | 面向 AI/ML 的专用处理器 | 通用并行计算加速器 | 通用计算与控制平面 |
| **设计目标** | 能效优先，数据流优化 | 算力优先，大规模并行 | 通用性优先，灵活调度 |
| **典型算力** | 中等（如 10-50 TOPS） | 高（如 50-200 TFLOPS） | 低（如 100-500 GFLOPS） |
| **能效比** | 最高（TOPS/W） | 中等 | 最低 |
| **功耗** | 低（< 5W） | 中高（5-20W） | 低（< 3W） |
| **散热要求** | 低 | 中高 | 低 |
| **内存带宽** | 高（片上 SRAM 优化） | 中高（专用显存） | 低（系统内存总线） |
| **算子支持** | 有限（依赖厂商） | 广泛（通用性强） | 最广（任意算子） |
| **算力利用率** | 高（算子匹配时） | 中高（需优化） | 低（受限于架构） |
| **适用场景** | 语音识别、低分辨率视觉、轻量 LLM | 大模型推理、高分辨率视觉、自定义算子 | 算子回退、控制逻辑、超轻量模型 |
| **软件栈依赖** | 强（厂商特定 SDK） | 中（OpenCL、Metal、Vulkan） | 弱（跨平台框架多） |
| **部署复杂度** | 中等 | 中等 | 低 |
| **端侧适配性** | 最佳（功耗、尺寸友好） | 中等（功耗、散热压力大） | 基础兜底 |

**工程启示**：

1. **优先 NPU，兜底 CPU，GPU 作为增强**：在满足功能的前提下，优先选择能效最高的 NPU；NPU 不支持时回退到 CPU；GPU 仅在算力不足或需要通用性时使用。
2. **避免全硬件满载**：NPU、GPU、CPU 同时满载会导致功耗飙升和设备过热，合理分配任务负载是关键。
3. **算子支持度是硬约束**：硬件选型不能只看峰值算力，算子支持度和软件栈成熟度同样重要。
4. **端侧推理是系统工程**：硬件选型需要结合功耗预算、散热条件、软件栈兼容性、部署成本和总拥有成本（TCO）综合决策。

## 端侧硬件选型的工程约束

在端侧部署推理系统时，硬件选型不仅是"算力越高越好"，还需考虑以下工程约束：

### 功耗预算

- **电池设备**：推理任务需控制在可接受的功耗范围，避免快速耗电。例如：
  - 持续运行推理任务 1 小时，电量消耗应 < 10%
  - 单次推理任务的功耗应 < 100mWh
- **固定供电设备**：功耗影响散热成本和可靠性，需平衡算力与热设计。例如：
  - 工控机的散热系统需要支持持续 20-50W 的推理负载
  - 车载设备需要符合汽车电子的功耗标准（如 ISO 26262）

### 散热条件

- **被动散热**：无风扇设备（如手机、IoT 终端）需严格控制发热，避免降频。例如：
  - 手机在 25℃ 环境温度下，持续推理 10 分钟后温度应 < 45℃
  - 温度超过 50℃ 时，触发热降频，性能下降 30-50%
- **主动散热**：有风扇设备（如工控机、车载终端）散热压力相对较小，但仍需避免持续高负载。例如：
  - 风扇转速超过 5000 RPM 时，噪音和可靠性问题凸显
  - 持续高负载会缩短风扇寿命

### 内存带宽

- **推理瓶颈常在内存**：尤其是大模型推理，KV Cache 和参数访问会占用大量带宽。例如：
  - 7B INT4 模型推理时，每生成一个 token 需要访问约 2GB 内存
  - 若内存带宽为 50 GB/s，则每秒最多生成 25 个 token
- **片上内存是关键**：NPU 的高能效往往依赖片上 SRAM，减少频繁内存访问。例如：
  - Apple Neural Engine 配备 16MB 片上 SRAM，可缓存关键层参数
  - 高通 Hexagon NPU 配备 4-8MB 片上 SRAM，用于缓存激活值

### 算力利用率

- **峰值算力≠实际算力**：模型结构、算子支持度、调度策略都会影响算力利用率。例如：
  - NPU 的峰值算力是 50 TOPS，但实际推理时利用率可能只有 30-60%
  - GPU 的峰值算力是 100 TFLOPS，但受限于显存带宽，实际利用率可能只有 20-40%
- **优化空间大**：通过 kernel 融合、算子替换、内存布局优化可显著提升利用率。例如：
  - 算子融合可以将 GPU 利用率从 30% 提升到 50%
  - 内存布局优化可以将 NPU 利用率从 40% 提升到 70%

### 异构调度复杂度

- **多硬件协作带来挑战**：数据移动、同步、优先级管理需要精细化调度。例如：
  - NPU 和 GPU 之间频繁切换会导致 10-50% 的性能下降
  - 异构调度框架的开发成本可能高达 3-6 人月
- **调度策略需定制**：不同设备的硬件组合、工作负载特征需要适配不同策略。例如：
  - 高端设备（NPU + GPU）：优先使用 NPU，GPU 作为增强
  - 中端设备（仅有 NPU）：只能使用 NPU，算子回退到 CPU
  - 低端设备（仅有 CPU）：只能使用 CPU，需要极致优化

### 总系统成本

- **硬件成本**：独立 GPU 成本高，NPU 和 iGPU 更适合大规模边缘部署。例如：
  - 独立 GPU（如 NVIDIA Jetson）成本约 $500-2000
  - 集成 NPU 的 SoC 成本约 $50-200
  - 仅 CPU 的 SoC 成本约 $10-50
- **开发成本**：不同硬件的适配工作量不同，软件栈兼容性影响开发效率。例如：
  - 适配 NPU 需要 2-4 人月（需要学习厂商 SDK）
  - 适配 GPU 需要 1-2 人月（OpenCL/Metal 相对成熟）
  - 适配 CPU 需要 0.5-1 人月（通用性最强）
- **运维成本**：散热、电源、可靠性等运维成本需纳入总拥有成本（TCO）计算。例如：
  - 散热系统成本可能占硬件成本的 10-20%
  - 电源冗余系统成本可能占硬件成本的 5-10%
  - 定期维护成本可能占 TCO 的 5-15%

---

## 总结

端侧推理工程师需要深入理解 NPU、GPU、CPU 的能力边界与互补关系：

- **NPU 是主力**：能效比最高，适合低功耗、低时延场景，但算子支持有限，需要适配不同厂商的 SDK
- **GPU 是增强**：算力强、通用性好，适合大模型和自定义算子，但功耗和散热压力大，需谨慎使用
- **CPU 是兜底**：通用性强、可用性高，但能效比低，适合轻量任务和算子回退

在实际部署中，硬件选型需综合考虑功耗预算、散热条件、内存带宽、算力利用率、异构调度复杂度和总系统成本。推理框架通过 Delegate 机制和动态调度，实现异构硬件的协同优化。

**记住**：端侧推理的核心目标不是追求峰值算力，而是在有限资源约束下实现可接受的性能和能耗平衡。硬件选型是系统工程，需要结合应用场景、设备能力、软件栈成熟度和总拥有成本综合决策。
