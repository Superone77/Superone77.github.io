# 第十章：端侧部署与更新

端侧大模型推理的部署与更新是产品化的关键环节。与云侧推理不同，端侧模型部署需要考虑设备资源限制、网络环境、版本管理、灰度发布等多方面因素。本章将系统介绍模型打包与分发、热更新机制、版本管理、A/B 测试与灰度发布的核心实践。

---

## 10.1 模型打包与分发

### 10.1.1 模型与应用打包方式

端侧模型部署通常有两种打包方式：静态打包和动态下载。

**静态打包**：将模型文件直接嵌入应用包中，随应用一起分发。这是最简单的方式，适用于模型较小（< 50MB）且更新频率低的场景。

- **iOS**：将 `.mlmodelc` 文件添加到 Xcode 项目中，编译时自动打包到应用 Bundle 中
- **Android**：将 `.tflite` 文件放入 `assets` 目录，随 APK 一起分发

**静态打包的优势**：
- 部署简单，无需额外网络请求
- 模型立即可用，无加载延迟
- 适合对离线可用性要求高的场景

**静态打包的劣势**：
- 应用体积增大，影响下载和安装
- 模型更新需要发布新版本应用，审核周期长
- 无法实现灰度发布和 A/B 测试

**动态下载**：模型文件托管在云端，应用首次运行或需要时动态下载。这是主流方式，适用于模型较大（> 50MB）或需要频繁更新的场景。

- **模型托管**：CDN（如阿里云 OSS、腾讯云 COS、AWS S3）
- **版本管理**：使用版本号或哈希值标识不同版本
- **下载策略**：首次运行下载、后台静默下载、按需下载

**动态下载的优势**：
- 应用体积小，下载和安装快
- 模型更新无需发布新版本应用
- 可实现灰度发布和 A/B 测试
- 可根据设备特征选择合适版本

**动态下载的劣势**：
- 首次使用需要下载，增加加载延迟
- 依赖网络，影响离线可用性
- 需要处理下载失败、网络中断等异常

### 10.1.2 模型文件大小优化

模型文件大小直接影响应用体积和下载时间。端侧推理通常使用量化、剪枝、蒸馏等技术压缩模型，但即使压缩后，大模型仍可能超过 100MB。以下是模型文件大小优化的策略。

**量化**：将模型权重从 FP32 量化为 INT8 或 INT4，可减少模型大小 4 倍或 8 倍。

- **权重量化**：减少权重内存占用，最常见的量化方式
- **激活量化**：减少推理时的计算量和内存带宽
- **混合精度**：关键层使用高精度（如 FP16），非关键层使用低精度（如 INT8）

**模型分片**：将大模型拆分为多个小文件，按需加载。

- **分层加载**：将模型按层分片，推理时逐层加载
- **分块加载**：将模型按参数块分片，推理时按需加载
- **权重压缩**：使用 ZIP、GZIP 等算法压缩模型文件

**模型格式优化**：选择合适的模型格式，减少文件大小。

- **TFLite**：支持 flatbuffer 格式，文件紧凑
- **ONNX**：支持多种压缩选项
- **CoreML**：使用 `.mlmodelc` 格式，优化存储

**案例**：一个 7B 参数的 LLM：
- FP32 格式：约 28GB（端侧不可行）
- INT8 量化：约 7GB（仍较大）
- INT4 量化：约 3.5GB（可接受）
- 使用模型分片后，每片约 500MB，可按需加载

### 10.1.3 动态模型下载机制

动态模型下载需要设计健壮的下载机制，处理网络中断、下载失败、版本管理等问题。

**下载流程**：
1. **版本检查**：应用启动或需要模型时，向服务端请求最新模型版本
2. **本地校验**：检查本地是否有可用模型，版本是否符合要求
3. **下载触发**：如果本地模型过旧或不存在，触发下载
4. **断点续传**：使用 HTTP Range 请求支持断点续传
5. **完整性校验**：下载完成后校验文件哈希（如 SHA256）
6. **解压与加载**：解压模型文件并加载到内存

**下载策略**：
- **同步下载**：阻塞界面，下载完成后才继续（适合首次启动）
- **异步下载**：后台下载，不阻塞界面（适合更新）
- **静默下载**：在 WiFi 环境下自动下载（推荐）
- **按需下载**：用户主动触发下载（适合可选模型）

**异常处理**：
- **网络中断**：自动重试，支持断点续传
- **下载失败**：降级使用旧版本或本地缓存
- **存储空间不足**：提示用户清理空间或使用旧版本
- **版本冲突**：服务端控制版本兼容性，客户端校验

### 10.1.4 CDN 与缓存策略

CDN（内容分发网络）加速模型下载，缓存策略减少重复下载。

**CDN 部署**：
- **全球分发**：使用 CDN 的全球节点，就近下载
- **压缩算法**：启用 GZIP、Brotli 等压缩算法
- **边缘缓存**：CDN 边缘节点缓存模型文件，减少源站压力
- **预加载**：在低峰期预加载模型到 CDN 节点

**缓存策略**：
- **强缓存**：使用 Cache-Control: max-age=31536000，长期缓存
- **版本化 URL**：在 URL 中加入版本号或哈希值（如 `model_v1.tflite?sha=abc123`）
- **ETag 校验**：使用 ETag 检查文件是否变化
- **本地缓存**：下载后缓存到本地磁盘，避免重复下载

**缓存失效**：
- **版本更新**：新版本发布后，更新 URL 中的版本号
- **强制刷新**：在紧急情况下，可强制客户端刷新缓存
- **灰度发布**：通过服务端控制，只对部分用户返回新版本

## 10.2 热更新机制

### 10.2.1 应用内模型更新

热更新是指无需发布新版本应用，直接在应用内更新模型文件的能力。端侧模型热更新需要考虑更新时机、更新策略、异常处理等问题。

**更新触发**：
- **主动更新**：服务端推送更新通知，客户端下载新模型
- **自动更新**：应用启动时自动检查并下载新模型
- **静默更新**：在 WiFi 环境下后台下载新模型
- **按需更新**：用户主动触发更新（如设置中"检查更新"）

**更新流程**：
1. **版本检查**：向服务端请求最新模型版本（版本号、下载 URL、哈希值）
2. **版本比对**：比对本地模型版本与服务端版本
3. **下载模型**：如果需要更新，下载新模型文件
4. **完整性校验**：下载完成后校验文件哈希
5. **原子替换**：确保模型替换是原子操作，避免替换一半导致损坏
6. **重启推理**：卸载旧模型，加载新模型，恢复推理服务

**原子替换策略**：
- **临时文件**：下载到临时文件（如 `model_v2.tflite.tmp`），校验通过后重命名
- **多文件共存**：新旧模型文件共存，切换时直接切换文件引用
- **数据库记录**：使用数据库记录当前激活的模型版本，支持回滚

### 10.2.2 灰度更新策略

灰度更新是指逐步将新模型发布给部分用户，观察效果后再全量发布。灰度更新可以降低风险，快速发现问题。

**灰度维度**：
- **设备特征**：设备型号、系统版本、内存大小等
- **用户分组**：用户 ID 分组、地区、使用频率等
- **随机抽样**：随机抽取一定比例的用户

**灰度策略**：
- **百分比灰度**：按百分比分配用户（如 1%、5%、10%、50%、100%）
- **设备灰度**：只对特定设备（如 iPhone 15）发布新模型
- **地区灰度**：只对特定地区（如中国、美国）发布新模型
- **实验组对比**：新模型作为实验组，旧模型作为对照组，对比指标

**灰度监控**：
- **下载成功率**：监控新模型下载成功率
- **加载成功率**：监控新模型加载成功率
- **推理性能**：监控推理延迟、内存占用
- **业务指标**：监控用户活跃度、转化率等业务指标
- **错误率**：监控推理错误率、崩溃率

### 10.2.3 更新失败处理

更新过程中可能出现各种异常，需要设计健壮的失败处理机制。

**常见失败场景**：
- **网络中断**：下载过程中网络断开
- **存储空间不足**：设备存储空间不足，无法保存新模型
- **下载文件损坏**：下载过程中文件损坏，哈希校验失败
- **加载失败**：模型文件下载成功，但加载失败（格式不支持、算子不支持）
- **性能下降**：新模型推理性能不如旧模型

**失败处理策略**：
- **自动重试**：网络中断时自动重试（如最多重试 3 次）
- **降级使用**：更新失败时降级使用旧版本或本地缓存
- **错误上报**：将失败信息上报到服务端，便于排查问题
- **用户提示**：关键更新失败时提示用户（如存储空间不足）
- **回滚机制**：新模型有问题时自动回滚到旧版本

### 10.2.4 回滚机制

回滚机制是指在新模型出现问题时，快速切换回旧版本的能力。回滚是灰度发布和 A/B 测试的重要保障。

**回滚触发**：
- **自动回滚**：监控到错误率飙升、性能下降等异常时自动回滚
- **手动回滚**：运营人员手动触发回滚
- **用户回滚**：用户反馈问题后，快速回滚到稳定版本

**回滚策略**：
- **多版本共存**：新旧模型文件共存，回滚时直接切换文件引用
- **版本记录**：记录每个设备使用的模型版本，支持精确回滚
- **批量回滚**：对灰度发布的设备批量回滚
- **选择性回滚**：只回滚有问题的设备或用户组

**回滚保障**：
- **数据一致性**：确保回滚后数据一致，不会出现版本混乱
- **原子切换**：确保回滚是原子操作，避免切换过程中出现问题
- **快速响应**：回滚应该能够在几分钟内完成，降低影响范围
- **事后分析**：回滚后分析问题原因，避免再次发生

## 10.3 版本管理

### 10.3.1 多版本共存管理

端侧推理可能需要同时维护多个模型版本，以支持不同的功能或场景。多版本共存管理需要解决版本隔离、资源占用、版本切换等问题。

**版本隔离**：
- **文件命名**：使用版本号或哈希值命名模型文件（如 `model_v1.tflite`、`model_v2.tflite`）
- **目录隔离**：将不同版本的模型放在不同目录（如 `/models/v1/`、`/models/v2/`）
- **配置隔离**：为每个版本配置独立的推理上下文（如 Runtime 实例）

**版本切换**：
- **软切换**：通过配置切换版本，无需重新加载模型
- **硬切换**：卸载旧模型，加载新模型（适合大版本更新）
- **灰度切换**：逐步切换部分用户到新版本

**资源占用**：
- **内存占用**：同时加载多个模型会增加内存占用，需要控制并发版本数量
- **存储占用**：多个版本文件会占用较多存储，需要定期清理
- **加载时间**：切换版本时需要加载新模型，会增加切换延迟

**使用场景**：
- **A/B 测试**：同时运行多个版本，对比效果
- **功能灰度**：新功能使用新模型，旧功能使用旧模型
- **兼容性**：旧设备使用轻量模型，新设备使用完整模型

### 10.3.2 版本兼容性处理

版本兼容性是指新旧版本之间的兼容性问题。如果新版本模型的输入输出格式、推理接口发生变化，可能导致旧版本应用无法使用新模型。

**兼容性类型**：
- **向后兼容**：新版本兼容旧版本的接口，旧版本应用可以使用新模型
- **向前兼容**：旧版本兼容新版本的接口，新版本应用可以使用旧模型
- **不兼容**：新旧版本互不兼容，需要强制升级

**兼容性设计**：
- **接口稳定**：保持推理接口稳定，避免频繁变化
- **版本协商**：客户端和服务端协商版本，选择兼容版本
- **降级策略**：新版本模型不可用时，降级使用旧版本
- **适配层**：在客户端实现适配层，兼容不同版本

**版本号规范**：
- **语义化版本号**：使用 `主版本.次版本.修订号` 格式（如 `1.2.3`）
- **主版本号**：不兼容的 API 变更（如 `1.0.0` → `2.0.0`）
- **次版本号**：向后兼容的功能性新增（如 `1.0.0` → `1.1.0`）
- **修订号**：向后兼容的问题修正（如 `1.0.0` → `1.0.1`）

### 10.3.3 版本降级策略

版本降级是指在新版本出现问题时，将设备回退到旧版本。版本降级是保障服务稳定的重要手段。

**降级触发**：
- **自动降级**：监控到异常时自动降级
- **手动降级**：运营人员手动触发降级
- **用户降级**：用户反馈问题后，运营人员降级

**降级策略**：
- **全局降级**：所有设备都降级到旧版本
- **部分降级**：只降级有问题的设备或用户组
- **选择性降级**：只降级特定版本的用户

**降级保障**：
- **可用性**：降级后服务应该立即可用，避免影响用户体验
- **数据一致性**：确保降级后数据一致，不会出现版本混乱
- **快速响应**：降级应该能够在几分钟内完成，降低影响范围
- **事后分析**：降级后分析问题原因，避免再次发生

**降级风险**：
- **功能缺失**：旧版本可能缺少新功能，影响用户体验
- **性能下降**：旧版本可能性能较差
- **兼容性问题**：旧版本可能与新版本应用不兼容

### 10.3.4 版本清理机制

版本清理是指定期清理不再使用的旧版本，释放存储空间。版本清理是长期维护的重要环节。

**清理策略**：
- **时间清理**：定期清理（如每月清理一次）
- **存储清理**：存储空间不足时清理
- **版本清理**：只保留最近 N 个版本（如最近 3 个版本）
- **灰度清理**：灰度发布完成后清理旧版本

**清理规则**：
- **删除旧版本**：删除不再使用的旧版本文件
- **保留最新版本**：保留最新版本的文件
- **保留稳定版本**：保留经过验证的稳定版本
- **删除临时文件**：删除下载过程中的临时文件

**清理触发**：
- **自动清理**：应用启动时检查并清理
- **手动清理**：用户在设置中手动清理
- **存储预警**：系统存储空间不足时清理

## 10.4 A/B 测试与灰度发布

### 10.4.1 端侧 A/B 测试设计

A/B 测试是指将用户分成多个组，每组使用不同的模型版本，对比效果。端侧 A/B 测试需要考虑分组策略、实验隔离、指标采集等问题。

**分组策略**：
- **随机分组**：随机分配用户到不同组（如 A 组、B 组）
- **特征分组**：根据用户特征分组（如设备型号、地区、使用频率）
- **哈希分组**：使用用户 ID 的哈希值分组，确保同一用户始终在同一组

**实验隔离**：
- **版本隔离**：不同组使用不同的模型版本
- **数据隔离**：不同组的推理数据分开存储和分析
- **结果隔离**：不同组的实验结果独立统计和分析

**实验设计**：
- **对照组**：使用旧模型，作为基准
- **实验组**：使用新模型，对比效果
- **多组对比**：可以设计多个实验组，对比不同策略

### 10.4.2 实验指标追踪

实验指标是评估模型效果的关键。端侧 A/B 测试需要采集多种指标，全面评估模型效果。

**性能指标**：
- **推理延迟**：平均推理时间、P50、P95、P99
- **内存占用**：模型加载后的内存占用
- **CPU 占用**：推理时的 CPU 占用率
- **功耗**：推理时的功耗

**质量指标**：
- **精度**：模型在任务上的精度（如准确率、F1 分数）
- **BLEU/ROUGE**：生成任务的质量指标
- **用户评分**：用户对生成结果的评分

**业务指标**：
- **用户活跃度**：DAU、MAU
- **转化率**：核心功能的转化率
- **留存率**：用户留存率
- **使用时长**：用户使用时长

**异常指标**：
- **错误率**：推理错误率
- **崩溃率**：应用崩溃率
- **超时率**：推理超时率

### 10.4.3 灰度发布策略

灰度发布是指逐步将新模型发布给部分用户，观察效果后再全量发布。灰度发布可以降低风险，快速发现问题。

**灰度阶段**：
- **小范围灰度**：先发布给 1%-5% 的用户，观察效果
- **中范围灰度**：如果效果良好，扩大到 10%-20% 的用户
- **大范围灰度**：继续扩大到 50%-80% 的用户
- **全量发布**：如果效果稳定，发布给 100% 的用户

**灰度策略**：
- **时间灰度**：逐步增加发布时间窗口
- **比例灰度**：逐步增加发布用户比例
- **地域灰度**：先在特定地区发布，再扩大到全球
- **设备灰度**：先在特定设备发布，再扩大到所有设备

**灰度监控**：
- **实时监控**：实时监控指标，及时发现异常
- **告警机制**：设置阈值，触发告警
- **快速回滚**：发现问题时快速回滚

### 10.4.4 实验终止与决策

实验终止是指结束 A/B 测试，做出最终决策。实验终止需要基于数据和经验，做出科学决策。

**终止条件**：
- **达到显著性**：实验指标达到统计学显著性（如 p < 0.05）
- **样本量充足**：实验样本量足够大（如每组至少 1000 个用户）
- **时间足够长**：实验时间足够长（如至少 7 天）
- **出现严重问题**：新模型出现严重问题，需要立即终止

**决策类型**：
- **发布新版本**：新模型效果显著优于旧模型，全量发布
- **保持旧版本**：新模型效果不明显，保持旧版本
- **优化后再发布**：新模型有潜力但需要优化，优化后再发布
- **终止实验**：新模型效果不佳，终止实验

**决策依据**：
- **数据驱动**：基于实验指标和数据做出决策
- **用户反馈**：结合用户反馈做出决策
- **成本收益**：评估发布新模型的成本和收益
- **风险评估**：评估发布新模型的风险
