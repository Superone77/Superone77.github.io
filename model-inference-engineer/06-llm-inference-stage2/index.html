端侧 LLM 推理链路进阶：Streaming、Session 与 KV Cache 生命周期 #

端侧 LLM 推理的核心挑战不仅在于单次推理的性能，更在于多轮对话的内存管理和用户感知延迟的优化。Streaming 输出、Session State 管理和 KV Cache 生命周期是端侧推理进阶优化的关键。

Streaming 输出与延迟优化 #

流式推理的优势 #

流式推理（Streaming）是指在推理过程中逐 token 生成并输出，而不是等待全部生成完成后一次性输出。流式推理的优势：

  • 降低用户感知延迟：第一个 token 的输出时间（TTFT）决定了用户感知的响应速度，流式推理可以在 Prefill 阶段完成后立即输出第一个 token
  • 提高用户体验：用户可以实时看到生成过程，减少等待焦虑
  • 支持长文本生成：对于长文本生成任务，流式推理可以让用户提前看到部分结果，即使生成未完成
  • 节省内存：不需要缓存全部生成结果，逐 token 输出后可释放内存

端侧推理的流式输出优势更明显：端侧设备通常网络不稳定，流式输出可以减少用户等待时间，提升用户体验。

分块输出策略 #

流式推理的分块输出策略直接影响用户感知延迟和性能：

1. 单 token 输出

  • 优点：延迟最低，用户最快看到第一个 token
  • 缺点：频繁输出导致网络传输开销大，增加 UI 刷新压力
  • 适用场景：网络稳定、UI 刷新开销小的端侧推理

2. 固定块大小输出

  • 优点：平衡延迟和性能，减少网络传输和 UI 刷新开销
  • 缺点：第一个 token 的输出时间略高于单 token 输出
  • 适用场景：网络不稳定、UI 刷新开销大的端侧推理
  • 推荐块大小：2-4 个 token（根据网络和 UI 能力调整）

3. 动态块大小输出

  • 优点：根据网络状况和生成速度动态调整块大小，优化用户体验
  • 缺点：实现复杂，需要实时监控网络和生成速度
  • 适用场景：网络波动大、生成速度不稳定的环境

端侧推理的分块输出策略需要综合考虑：

  • 网络状况：网络不稳定时使用固定块大小，网络稳定时使用单 token 输出
  • UI 刷新开销：UI 刷新开销大时使用固定块大小，UI 刷新开销小时使用单 token 输出
  • 生成速度：生成速度快时使用大块大小，生成速度慢时使用小块大小

用户感知延迟优化 #

用户感知延迟（Perceived Latency）是流式推理的关键指标，优化用户感知延迟需要从多个维度考虑：

1. 首字延迟（TTFT）优化

  • 缩短系统提示：系统提示越短，Prefill 阶段时间越短，TTFT 越小
  • 优化 Prefill 阶段：使用 NPU/GPU 进行 Prefill 阶段，提升计算效率
  • 异步 Prefill：用户输入到达前，提前 Prefill，减少用户感知延迟
  • 热点缓存：缓存常用系统提示的 KV Cache，避免重复计算

2. 分块输出优化

  • 选择合适的块大小：根据网络状况和 UI 刷新开销选择合适的块大小
  • 动态调整块大小：根据网络状况和生成速度动态调整块大小
  • 预输出部分内容：在 Prefill 阶段完成后，先输出系统提示等固定内容

3. UI 优化

  • 异步渲染：使用异步渲染，避免阻塞主线程
  • 增量渲染：使用增量渲染，避免全量刷新
  • 动画过渡：使用动画过渡，平滑 token 生成过程，减少用户感知延迟

4. 用户交互优化

  • 提前加载模型：用户交互前提前加载模型，减少响应时间
  • 预测用户输入：根据上下文预测用户输入，提前 Prefill
  • 缓存对话上下文：缓存对话上下文，减少重复计算

服务端与客户端流式配合 #

对于端云协同场景，服务端与客户端的流式配合需要考虑：

1. 服务端流式输出

  • 服务端逐 token 生成：服务端逐 token 生成并输出，减少服务端延迟
  • 服务端压缩输出：服务端压缩输出，减少网络传输开销
  • 服务端缓存 KV Cache：服务端缓存 KV Cache，支持多轮对话

2. 客户端流式接收

  • 客户端逐 token 接收：客户端逐 token 接收并渲染，减少客户端延迟
  • 客户端缓存接收：客户端缓存接收结果，支持离线查看
  • 客户端反馈机制：客户端实时反馈生成进度，提升用户体验

3. 端云协同流式

  • 端侧优先流式：优先使用端侧推理，流式输出；端侧推理失败时降级到云侧流式
  • 云侧流式降级：云侧推理流式输出时，降级为批量输出，减少网络传输开销
  • 端云混合流式：端侧处理简单请求，云侧处理复杂请求，混合流式输出

端侧推理的流式输出优化需要综合考虑端侧设备性能、网络状况、UI 能力等因素，找到最佳平衡点。

Session State 与多轮对话 #

会话状态的定义与管理 #

会话状态（Session State）是多轮对话的核心概念，定义了：

  • 当前会话的上下文：包括系统提示、历史消息、当前用户输入
  • KV Cache 状态：Prefill 阶段生成的 KV Cache，用于多轮对话的复用
  • 会话生命周期：会话的创建、激活、挂起、销毁

端侧推理的会话状态管理需要考虑：

  • 内存限制：端侧设备内存有限，需要限制并发会话数
  • 性能优化：会话切换时需要高效加载和卸载 KV Cache
  • 隐私保护：会话状态可能包含敏感信息，需要加密存储

会话状态的定义

class SessionState:
    session_id: str              # 会话唯一标识
    system_prompt: str           # 系统提示
    messages: List[Message]       # 历史消息列表
    kv_cache: KVCache            # KV Cache 状态
    created_at: float            # 会话创建时间
    last_active_at: float        # 会话最后活跃时间
    is_active: bool              # 会话是否活跃

会话状态的管理

  • 会话创建：用户首次交互时创建新会话，初始化系统提示和历史消息
  • 会话激活：用户切换到已有会话时，激活会话，加载 KV Cache
  • 会话挂起：用户离开会话时，挂起会话，保留 KV Cache，释放部分资源
  • 会话销毁：会话超时或用户主动删除时，销毁会话，释放 KV Cache

多轮对话的 KV Cache 复用 #

多轮对话的 KV Cache 复用是端侧推理的性能关键：

KV Cache 复用的优势

  • 避免重复计算：多轮对话的 Prefill 阶段可以复用上一轮的 KV Cache，避免重复计算
  • 降低内存占用：复用 KV Cache 不需要重新生成，减少内存占用
  • 提升响应速度：复用 KV Cache 可以直接进入 Decode 阶段，提升响应速度

KV Cache 复用的实现

  • 增量扩展 KV Cache：每轮对话时，将新的 token 添加到 KV Cache，而不是重新生成整个 KV Cache
  • KV Cache 分片存储：将 KV Cache 分片存储，按需加载和卸载
  • KV Cache 压缩：对 KV Cache 进行压缩（如 INT8 量化），减少内存占用

KV Cache 复用的挑战

  • KV Cache 管理复杂：多轮对话的 KV Cache 需要高效管理，避免内存泄漏
  • KV Cache 版本控制：多轮对话的 KV Cache 可能需要版本控制，支持回滚
  • KV Cache 并发访问：多个会话并发访问 KV Cache 时，需要线程安全

端侧推理的 KV Cache 复用需要综合考虑内存限制、性能要求、实现复杂度等因素。

会话的生命周期 #

会话的生命周期包括创建、激活、挂起、销毁四个阶段：

1. 会话创建

  • 创建时机：用户首次交互时创建新会话
  • 初始化内容：初始化系统提示、空的历史消息、空的 KV Cache
  • 资源分配：分配内存、加载 tokenizer、初始化推理引擎

2. 会话激活

  • 激活时机：用户切换到已有会话时激活会话
  • 加载 KV Cache：加载会话的 KV Cache，准备推理
  • 更新活跃时间：更新会话的最后活跃时间

3. 会话挂起

  • 挂起时机：用户离开会话时挂起会话
  • 保留 KV Cache：保留 KV Cache，但不占用计算资源
  • 释放部分资源：释放部分计算资源（如 GPU/NPU 占用）

4. 会话销毁

  • 销毁时机：会话超时或用户主动删除时销毁会话
  • 释放 KV Cache：释放 KV Cache，释放内存
  • 释放资源：释放所有相关资源（如 tokenizer、推理引擎）

端侧推理的会话生命周期管理需要考虑：

  • 内存限制：限制并发会话数，避免 OOM
  • 性能优化：会话切换时高效加载和卸载 KV Cache
  • 用户体验：会话切换时平滑过渡，避免卡顿

会话并发与隔离 #

端侧推理的会话并发与隔离需要考虑：

1. 会话并发策略

  • 单会话优先：端侧设备资源有限，优先支持单会话，多会话时串行处理
  • 多会话并发：如果内存和计算资源充足，支持多会话并发，提升用户体验
  • 动态调整并发数：根据资源占用动态调整并发会话数，避免 OOM

2. 会话隔离策略

  • 内存隔离：每个会话独立分配内存，避免相互干扰
  • 计算资源隔离：每个会话独立分配计算资源（如 GPU/NPU 占用）
  • KV Cache 隔离：每个会话独立管理 KV Cache，避免冲突

3. 会话优先级

  • 活跃会话优先：优先分配资源给活跃会话，提升用户体验
  • 新会话降级：新会话创建时，如果资源不足，可以降级或延迟创建
  • 超时会话销毁：超时会话自动销毁，释放资源

端侧推理的会话并发与隔离需要在性能、用户体验、资源限制之间找到平衡。

KV Cache 生命周期管理 #

KV Cache 的创建与初始化 #

KV Cache 的创建与初始化是端侧推理的第一步：

KV Cache 的结构 KV Cache 是自回归推理过程中缓存的前序 token 的 Key 和 Value 矩阵，用于加速后续推理：

class KVCache:
    keys: List[Tensor]          # 每层的 Key 矩阵 [num_layers, seq_len, hidden_dim]
    values: List[Tensor]        # 每层的 Value 矩阵 [num_layers, seq_len, hidden_dim]
    seq_len: int                # 当前序列长度
    max_seq_len: int            # 最大序列长度

KV Cache 的创建时机

  • 会话创建时：会话创建时创建空的 KV Cache，初始化 seq_len = 0
  • Prefill 阶段：Prefill 阶段一次性处理输入文本，生成完整的 KV Cache
  • 多轮对话时：每轮对话时增量扩展 KV Cache，而不是重新生成

KV Cache 的初始化

  • 分配内存：根据最大序列长度（如 4096）分配 KV Cache 内存
  • 初始化张量：初始化 Key 和 Value 张量为空张量
  • 设置最大序列长度：根据模型上下文长度设置最大序列长度

端侧推理的 KV Cache 创建与初始化需要考虑：

  • 内存限制：根据端侧设备内存限制调整最大序列长度
  • 量化策略：使用 INT8 量化减少 KV Cache 内存占用
  • 分片存储：将 KV Cache 分片存储，按需加载和卸载

每轮对话的 KV Cache 扩展 #

每轮对话时，KV Cache 需要增量扩展：

KV Cache 扩展过程

  1. Prefill 新输入：Prefill 阶段处理新的用户输入，生成新的 KV Cache 块
  2. 合并 KV Cache：将新的 KV Cache 块合并到现有 KV Cache
  3. 更新序列长度：更新 seq_len，记录新的序列长度

KV Cache 扩展的优化

  • 增量扩展：只扩展新的 token 的 KV Cache，不重新生成整个 KV Cache
  • 异步扩展：异步扩展 KV Cache，避免阻塞用户交互
  • 批量扩展：如果用户输入很长，分块扩展 KV Cache，避免阻塞 UI 线程

KV Cache 扩展的挑战

  • 内存占用增长：每轮对话 KV Cache 内存占用增长，需要限制最大序列长度
  • 性能开销：KV Cache 扩展需要内存拷贝，可能影响性能
  • 版本控制：多轮对话的 KV Cache 可能需要版本控制，支持回滚

端侧推理的 KV Cache 扩展需要在性能、内存占用、实现复杂度之间找到平衡。

KV Cache 的清理与回收 #

KV Cache 的清理与回收是端侧推理的内存管理关键：

KV Cache 清理时机

  • 会话销毁时：会话销毁时清理 KV Cache，释放内存
  • KV Cache 超长时：KV Cache 超过最大序列长度时，清理最早的 KV Cache 块（滑动窗口）
  • 内存不足时：内存不足时，清理非活跃会话的 KV Cache，释放内存

KV Cache 回收策略

  • 直接释放：直接释放 KV Cache 内存，立即回收
  • 内存池复用：使用内存池复用 KV Cache 内存，减少内存分配/释放开销
  • 延迟回收：延迟回收 KV Cache 内存，减少内存抖动

KV Cache 清理的优化

  • 按需清理：按需清理 KV Cache，避免过度清理影响性能
  • 热点保留：保留热点 KV Cache（如系统提示的 KV Cache），避免重复计算
  • 异步清理：异步清理 KV Cache，避免阻塞用户交互

端侧推理的 KV Cache 清理与回收需要在内存释放、性能、用户体验之间找到平衡。

多会话的 KV Cache 管理 #

多会话的 KV Cache 管理是端侧推理的复杂场景：

多会话 KV Cache 管理策略

  • 独立管理：每个会话独立管理 KV Cache，避免相互干扰
  • 共享部分 KV Cache：多个会话可以共享部分 KV Cache（如系统提示的 KV Cache）
  • KV Cache 池化：使用 KV Cache 池化机制，复用 KV Cache 内存

多会话 KV Cache 的挑战

  • 内存占用增长：多个会话同时存在时，KV Cache 内存占用增长，需要限制并发会话数
  • 性能竞争：多个会话并发推理时，竞争计算资源（如 GPU/NPU），影响性能
  • 管理复杂度：多会话的 KV Cache 管理复杂度增加，需要高效的数据结构

多会话 KV Cache 的优化

  • 限制并发数：根据内存和计算资源限制并发会话数，避免 OOM
  • 优先级调度：优先调度活跃会话，提升用户体验
  • KV Cache 分片：将 KV Cache 分片存储，按需加载和卸载，减少单次内存占用

端侧推理的多会话 KV Cache 管理需要在性能、用户体验、资源限制之间找到平衡。

总结 #

端侧 LLM 推理链路的进阶优化需要综合考虑 Streaming 输出、Session State 管理和 KV Cache 生命周期：

  • Streaming 输出优化：分块输出策略、用户感知延迟优化、端云协同流式配合
  • Session State 管理：会话状态定义、KV Cache 复用、会话生命周期、会话并发与隔离
  • KV Cache 生命周期管理：创建与初始化、每轮对话扩展、清理与回收、多会话管理

在实际部署中，端侧 LLM 推理的进阶优化需要根据硬件能力、软件栈兼容性、用户偏好综合调整。推理框架通过优化的 Streaming、Session State 和 KV Cache 管理，提供高效的端侧多轮对话能力。

记住：端侧 LLM 推理的核心目标不是追求最快的 TTFT 或最高的吞吐量，而是在有限资源约束下实现可接受的性能和质量平衡，同时提供流畅的多轮对话体验。
