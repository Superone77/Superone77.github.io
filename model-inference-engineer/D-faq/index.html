附录 D：常见问题 FAQ #

本附录汇总了端侧大模型推理、模型压缩与蒸馏、平台适配、安全与隐私等领域的常见问题及解答，帮助读者快速找到答案。

D.1 端侧推理常见问题 #

D.1.1 基础概念 #

Q1：什么是端侧推理？它与云侧推理有什么区别？

A：端侧推理（Edge Inference）指在用户设备（如手机、平板、智能音箱、车载系统）上直接进行模型推理，无需将数据发送到云端服务器。云侧推理（Cloud Inference）指在云端服务器上进行模型推理，用户设备只负责数据采集和结果展示。

区别：

  • 延迟：端侧推理延迟低（10-500ms），云侧推理延迟高（50-2000ms）
  • 隐私：端侧推理数据不出设备，隐私保护更好
  • 成本：端侧推理无需云端算力，成本更低
  • 能力：云侧推理算力充足，支持大模型和复杂任务

Q2：什么时候应该选择端侧推理？什么时候应该选择云侧推理？

A：选择端侧推理的场景：

  • 对延迟要求极高（<100ms）
  • 对隐私要求极高（数据不出设备）
  • 需要离线可用（无网络依赖）
  • 成本敏感（用户量大）

选择云侧推理的场景：

  • 需要大模型能力（>7B 参数）
  • 需要长上下文（>4096 tokens）
  • 需要复杂推理（多步推理、外部知识库）
  • 用户设备算力弱

Q3：端侧推理的主要挑战是什么？

A：端侧推理的主要挑战：

  • 模型容量约束：端侧设备内存有限，模型参数通常在 1B-7B 之间
  • 算力约束：推理速度依赖 NPU/GPU 算力，可能无法支持复杂模型
  • 模型更新成本：通过应用商店更新或内嵌下载机制，更新周期长
  • 多设备适配：不同设备的 NPU 能力和算子支持度差异大

D.1.2 性能优化 #

Q4：如何降低端侧推理的延迟？

A：降低端侧推理延迟的方法：

  • 量化：使用 INT8 量化，减少计算量（可降低 50-70% 延迟）
  • KV Cache 优化：减少 KV Cache 大小，使用量化 KV Cache
  • 硬件后端选择：优先使用 NPU，其次是 GPU
  • 模型压缩：使用剪枝、蒸馏等技术，减少模型大小
  • 投机采样：用小模型预测大模型输出，加速推理

Q5：如何在端侧设备上运行 7B 参数的模型？

A：在端侧设备上运行 7B 参数模型的方法：

  • 量化：使用 INT8 量化，将模型大小从 28GB（FP32）降低到 7GB（INT8）
  • KV Cache 优化：减少 KV Cache 大小，使用量化 KV Cache（INT8）
  • 模型分层：将模型分层，分时加载和推理
  • 内存管理：使用内存池，减少内存碎片
  • 硬件选择：选择 NPU 算力强、内存大的设备（如 iPhone 15 Pro、Samsung Galaxy S24）

Q6：如何评估端侧推理的性能？

A：评估端侧推理性能的指标：

  • 延迟：TTFT、生成延迟、端到端延迟
  • 吞吐量：TPS、QPS
  • 内存：模型内存占用、峰值内存占用
  • 功耗：功耗、能效比、电池续航
  • 准确率：Top-1 Accuracy、Top-5 Accuracy、F1 Score、Perplexity

使用工具：

  • Android：Android Profiler、Battery Historian
  • iOS：Xcode Instruments
  • 跨平台：ONNX Runtime Profiler、TFLite Profiler

D.1.3 端云协同 #

Q7：什么是端云协同推理？它有什么优势？

A：端云协同推理（Edge-Cloud Collaborative Inference）指在端侧设备和云端服务器之间合理分配计算任务，根据任务复杂度、网络状况、隐私要求等因素动态选择推理位置。

优势：

  • 灵活：根据任务复杂度选择推理位置
  • 高效：简单任务端侧处理，复杂任务云端处理
  • 隐私：高敏感度数据优先端侧处理
  • 成本：平衡端侧和云端成本

Q8：如何实现端云协同推理？

A：实现端云协同推理的方法：

  • 能力协商：端侧设备上报算力和模型版本，云端决定推理策略
  • 结果缓存：云端推理结果缓存到端侧，避免重复请求
  • 复杂请求云端处理：长对话、复杂推理、多模态任务发送到云端
  • 云端预热：提前加载模型和数据，减少推理延迟

D.2 模型压缩与蒸馏常见问题 #

D.2.1 量化 #

Q9：什么是量化？为什么要量化？

A：量化（Quantization）指将模型参数从高精度（如 FP32、FP16）转换为低精度（如 INT8、INT4），以减少模型大小和计算量。

量化的好处：

  • 模型大小减少：FP32 → INT8，模型大小减少 4 倍
  • 推理速度提升：INT8 计算速度比 FP32 快 2-4 倍
  • 内存占用减少：INT8 内存占用是 FP32 的 1/4
  • 功耗降低：INT8 计算功耗比 FP32 低 50-70%

Q10：量化的精度损失有多大？如何减少精度损失？

A：量化的精度损失取决于模型和任务：

  • 典型精度损失：Top-1 Accuracy 降低 1-5%，F1 Score 降低 1-3%
  • 减少精度损失的方法：
    • 使用感知量化（QAT）：在训练过程中模拟量化操作
    • 混合精度：关键层（如 Attention 层）保持高精度（FP16）
    • 优化校准：使用 Entropy 校准或 Percentile 校准
    • 调整温度参数：选择合适的温度参数 T（通常在 3-10 之间）

Q11：PTQ 和 QAT 有什么区别？应该选择哪种？

A：PTQ（Post-Training Quantization）和 QAT（Quantization-Aware Training）的区别：

特性 PTQ QAT
训练过程 不需要重新训练模型 需要在训练过程中模拟量化操作
训练成本 低（无需训练） 高（需要重新训练）
精度损失 较大（1-5%） 较小（<1%）
适用场景 模型训练完成后量化 对精度要求高的场景

选择建议：

  • 如果精度损失可接受（<3%），选择 PTQ（简单、快速）
  • 如果精度损失不可接受（>3%），选择 QAT（精度更高）

Q12：如何选择 W8A8 和 W4A16？

A：W8A8 和 W4A16 是两种常见的混合精度配置：

配置 权重量化 激活量化 模型大小 推理速度 精度损失
W8A8 INT8 INT8 小 快 中等
W4A16 INT4 FP16 更小 中等 较大

选择建议：

  • 如果对精度要求高（精度损失 <2%），选择 W8A8
  • 如果对模型大小要求高（模型大小 <4GB），选择 W4A16

D.2.2 剪枝 #

Q13：什么是剪枝？剪枝有什么好处？

A：剪枝（Pruning）指删除模型中不重要的参数或层，以减少模型大小和计算量。

剪枝的好处：

  • 模型大小减少：剪枝 50%，模型大小减少 50%
  • 推理速度提升：稀疏模型推理速度可提升 20-50%（需要硬件支持）
  • 内存占用减少：剪枝后的模型内存占用更小
  • 计算量减少：稀疏模型计算量减少 30-70%

Q14：结构化剪枝和非结构化剪枝有什么区别？

A：结构化剪枝和非结构化剪枝的区别：

特性 结构化剪枝 非结构化剪枝
剪枝粒度 删除整个通道、层或张量 删除权重中的单个元素
计算量减少 显著（需要硬件支持） 较小（需要稀疏计算支持）
精度损失 较大 较小
硬件要求 需要硬件支持稀疏计算 需要稀疏存储和计算支持

选择建议：

  • 如果硬件支持稀疏计算（如某些 NPU），选择结构化剪枝
  • 如果硬件不支持稀疏计算，选择非结构化剪枝

D.2.3 知识蒸馏 #

Q15：什么是知识蒸馏？为什么要蒸馏？

A：知识蒸馏（Knowledge Distillation）指将教师模型（大模型）的知识迁移到学生模型（小模型），以压缩模型大小或加速推理。

蒸馏的好处：

  • 模型大小减少：学生模型大小是教师模型的 1/10 到 1/100
  • 推理速度提升：学生模型推理速度是教师的 10-100 倍
  • 精度保留：学生模型可以保留教师模型 85-95% 的能力

Q16：如何选择教师模型和学生模型？

A：选择教师模型和学生模型的原则：

教师模型选择：

  • 能力匹配：教师模型的能力应覆盖端侧应用场景
  • 规模合理：教师模型过大（如 175B）会导致蒸馏成本高，过小（如 7B）会导致蒸馏效果差
  • 格式兼容：教师模型应易于转换为端侧支持的格式（如 ONNX、TFLite）
  • 开源友好：选择权重和代码开源的模型（如 LLaMA、ChatGLM）

学生模型选择：

  • 规模合适：学生模型大小应在端侧设备内存限制内（如 7B、3B）
  • 结构相似：学生模型结构应与教师模型相似，便于蒸馏
  • 算子支持：学生模型算子应在端侧硬件上有实现

Q17：知识蒸馏的损失函数如何设计？

A：知识蒸馏的损失函数通常包含两部分：

  1. 蒸馏损失（Distillation Loss）：学生模型输出与教师模型输出的差异
  2. 学生损失（Student Loss）：学生模型输出与真实标签的差异

总损失 = α × 蒸馏损失 + (1-α) × 学生损失

参数选择：

  • 温度参数 T：控制软标签的平滑度，通常在 3-10 之间，常见选择 T=5.0
  • 权重参数 α：平衡蒸馏损失和学生损失，通常在 0.5-0.9 之间，常见选择 α=0.7

D.3 平台适配常见问题 #

D.3.1 iOS 平台 #

Q18：在 iOS 上使用 CoreML 还是 ONNX Runtime？

A：选择 CoreML 或 ONNX Runtime 的原则：

特性 CoreML ONNX Runtime
集成难度 低（Apple 生态） 中（第三方库）
性能 高（Neural Engine 优化） 中（需要配置 EP）
硬件支持 CPU、GPU、Neural Engine CPU、GPU（Metal EP）
模型格式 CoreML 模型 ONNX 模型

选择建议：

  • 如果模型已经在 Apple 生态中，选择 CoreML
  • 如果需要跨平台，选择 ONNX Runtime

Q19：如何在 iOS 上优化 CoreML 模型性能？

A：在 iOS 上优化 CoreML 模型性能的方法：

  • 使用 Neural Engine：CoreML 自动选择最优计算设备（Neural Engine > GPU > CPU）
  • 模型量化：使用 CoreML Tools 进行 FP16 或 INT8 量化
  • 模型压缩：使用 CoreML Tools 压缩模型大小
  • 批量推理：如果适用，使用批量推理提升吞吐量
  • 异步加载：异步加载模型，避免阻塞主线程

Q20：如何在 iOS 上处理 CoreML 模型加密？

A：在 iOS 上处理 CoreML 模型加密的方法：

  • 使用 CoreML Tools 加密：coremltools.utils.encrypt_model(model, key=key_path)
  • 密钥管理：使用 Keychain 存储加密密钥
  • 加载加密模型：CoreML 自动解密模型并加载到内存中
  • 安全级别：使用 Complete Protection（设备锁定时无法访问）

D.3.2 Android 平台 #

Q21：在 Android 上使用 TFLite 还是 ONNX Runtime？

A：选择 TFLite 或 ONNX Runtime 的原则：

特性 TFLite ONNX Runtime
集成难度 低（Google 官方） 中（第三方库）
性能 高（NNAPI、XNNPU、GPU Delegate） 中（需要配置 EP）
硬件支持 CPU、GPU、NNAPI、Vendor NPU CPU、GPU、NNAPI
模型格式 TFLite 模型 ONNX 模型

选择建议：

  • 如果模型已经在 TensorFlow 生态中，选择 TFLite
  • 如果需要跨平台，选择 ONNX Runtime

Q22：如何在 Android 上优化 TFLite 模型性能？

A：在 Android 上优化 TFLite 模型性能的方法：

  • 使用硬件加速：启用 NNAPI、XNNPU、GPU Delegate
  • 模型量化：使用 TFLite Quantization 进行 INT8 量化
  • 模型压缩：使用 TFLite Model Compression 压缩模型大小
  • 批量推理：如果适用，使用批量推理提升吞吐量
  • 多线程：使用多线程推理，利用多核 CPU

Q23：如何在 Android 上处理 TFLite 模型保护？

A：在 Android 上处理 TFLite 模型保护的方法：

  • 模型混淆：使用工具（如 ProGuard）混淆模型结构
  • 模型加密：使用 Java Cryptography Architecture 加密模型文件
  • 密钥管理：使用 Android Keystore 存储加密密钥
  • 硬件安全模块：使用 StrongBox（HSM）存储密钥

D.4 安全与隐私常见问题 #

D.4.1 模型加密 #

Q25：为什么要加密模型？加密会影响推理性能吗？

A：加密模型的原因：

  • 保护知识产权：防止模型被逆向和窃取
  • 防止篡改：防止模型被篡改或植入后门
  • 合规要求：某些行业（如金融、医疗）要求模型加密

加密对推理性能的影响：

  • 模型加载时间：加密模型需要解密，加载时间可能增加 10-50%
  • 推理性能：解密后的模型在内存中运行，推理性能不受影响
  • 内存占用：加密模型和解密后模型同时存在于内存中，内存占用可能增加 10-20%

Q26：如何选择模型加密算法？

A：选择模型加密算法的原则：

算法 密钥长度 加密速度 安全级别 适用场景
AES-128 128 位 快 高 一般场景
AES-256 256 位 中 很高 高安全要求场景
RSA 2048/4096 位 慢 很高 密钥交换

选择建议：

  • 一般场景使用 AES-256
  • 密钥交换使用 RSA（配合 AES 使用）

D.4.2 数据隐私 #

Q27：端侧推理天然满足 GDPR 要求吗？

A：端侧推理在数据不出设备的情况下，天然满足 GDPR 的部分要求：

  • ✅ 数据最小化：数据在本地处理，不上传到云端
  • ✅ 被遗忘权：用户可以删除本地数据
  • ✅ 数据可移植权：用户可以导出本地数据
  • ⚠️ 知情权：需要向用户说明数据处理方式
  • ⚠️ 同意权：需要获取用户同意

建议：

  • 即使数据不出设备，也应该向用户说明数据处理方式
  • 应该获取用户同意，特别是敏感数据（如医疗数据）

Q28：如何识别和处理敏感数据？

A：识别和处理敏感数据的方法：

敏感数据识别：

  • 正则表达式：使用正则表达式匹配敏感数据模式（如身份证号、银行卡号）
  • 命名实体识别（NER）：使用 NLP 模型识别敏感实体（如人名、地址）
  • 规则引擎：使用预定义的规则识别敏感数据

敏感数据处理：

  • 替换：将敏感数据替换为占位符（如 [PHONE]、[ID_CARD]）
  • 哈希：使用哈希函数对敏感数据进行哈希处理
  • 加密：使用加密算法对敏感数据进行加密

D.4.3 安全防护 #

Q29：如何防止模型被逆向工程？

A：防止模型被逆向工程的方法：

  • 模型加密：加密模型文件，防止模型被导出
  • 模型混淆：混淆模型结构和算子名称
  • 代码混淆：混淆应用代码，防止应用被逆向
  • 二进制加固：加固编译后的二进制文件
  • 反调试：检测调试器，防止动态分析
  • 反模拟器：检测模拟器，防止在模拟器上分析

Q30：如何防止对抗攻击？

A：防止对抗攻击的方法：

  • 对抗训练：在训练数据中加入对抗样本，提高模型鲁棒性
  • 输入净化：对输入数据进行净化处理，移除对抗性扰动
  • 模型集成：使用多个模型进行集成预测，降低单个模型被攻击的风险
  • 异常检测：使用异常检测模型检测对抗样本

D.5 其他常见问题 #

D.5.1 项目管理 #

Q31：如何管理端侧推理项目？

A：管理端侧推理项目的方法：

  • 制定项目计划：明确项目目标、里程碑、交付物
  • 建立质量标准：制定性能指标、准确率指标、功耗指标
  • 自动化测试：建立自动化测试流程，定期测试性能和准确率
  • 版本管理：使用 Git 管理代码和模型版本
  • 文档管理：维护技术文档、API 文档、用户文档

Q32：如何持续优化端侧推理性能？

A：持续优化端侧推理性能的方法：

  • 性能监控：监控推理性能指标（延迟、吞吐量、内存、功耗）
  • 瓶颈定位：使用性能分析工具定位性能瓶颈
  • 优化迭代：根据瓶颈定位结果，优化模型、算法、硬件
  • A/B 测试：使用 A/B 测试验证优化效果
  • 用户反馈：收集用户反馈，优化用户体验

D.5.2 团队协作 #

Q33：端侧推理团队如何协作？

A：端侧推理团队的协作方法：

  • 明确分工：算法工程师负责模型优化，工程工程师负责工程实现，测试工程师负责质量保证
  • 定期同步：定期召开同步会议，讨论项目进展和问题
  • 知识共享：维护知识库，分享技术文章和实践经验
  • 代码审查：进行代码审查，保证代码质量
  • 文档共享：使用协作工具（如 Confluence、Notion）共享文档
